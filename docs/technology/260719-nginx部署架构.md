# Nginx 部署架构（多项目网关模式）

> 状态：实施中
> 日期：2026-07-19
> 域名：`airise.site`（腾讯云 / DNSPod）
> 范围：独立网关容器 `airise-gateway`（仓库 `nginx/` 自包含镜像），承载多个前后端分离项目（首个接入：PixelPack）。**不与 ripro / airc-nginx 共用容器**。

---

## 1. 背景与目标

服务器上有多个前后端分离项目（PixelPack、ripro-v5-pro、未来更多），都需要：HTTPS / WSS、文件上传下载与预览、SPA 静态托管。早期 PixelPack 采用「内层 nginx（pixelpack-web）+ 外层网关」的**双重 nginx** 结构，导致 WS/SSE/上传参数要在两层各配一遍，已踩坑（信令 `signal` 卡 pending）。

目标：建立**统一的网关托管模式**，所有项目一致接入，消除内层 nginx。

---

## 2. 现状分析

服务器上既有的 `airc-nginx` 对 ripro 项目**已采用规范的网关直发模式**（本架构以此为设计参照，但**新建独立容器 `airise-gateway`，不复用 airc-nginx**）：

| 挂载（ripro） | 用途 |
|---|---|
| `/opt/ripro-v5-pro/frontend/dist` → `/usr/share/nginx/html` (ro) | ripro SPA，网关直发 |
| `ripro-v5-pro_uploads_data` 卷 → `/var/www/uploads` (ro) | ripro 上传文件，网关直发 |
| `/etc/letsencrypt` → `/etc/letsencrypt` (ro) | certbot 证书 |
| `airc.conf` → `conf.d/airc.conf` (ro) | 单站点配置 |

ripro 模式即目标模式。PixelPack 此前的问题：

1. 🔴 **双重 nginx**：`airc-nginx → pixelpack-web(nginx) → api`，内层多余，WS/SSE 配置需双份。
2. 🟡 容器互联用 `172.17.0.1`（docker0 网桥 IP），脆弱。
3. 🟡 静态/上传走后端（FastAPI StaticFiles），不如 nginx 直发高效。
4. 🟡 多站点 conf.d 是单文件挂载，扩展性差。

---

## 3. 目标架构

```
                    Internet (HTTPS / WSS)
                          │
              ┌───────────▼───────────┐
              │  airise-gateway (网关) │  独立容器，仓库 nginx/ 自包含镜像
              │  · 唯一入口：TLS 终结 + 路由 + 直发静态
              │  · 通配证书 *.airise.site
              │  · conf.d/ snippets/ 构建期烤进镜像
              └───────────┬───────────┘
                          │ docker network: airise-web (external, 共享)
            ┌─────────────┼─────────────┬──────────────┐
            ▼             ▼             ▼              ▼
      pixelpack-api   ripro-backend   app3            appN
      (容器名直连，不再暴露宿主机端口、不再用 172.17.0.1)

  宿主路径（compose 挂载，网关 ro 直发）：
    /opt/<project>/web/dist    → /var/www/<project>          (SPA 静态)
    /opt/<project>/data/uploads→ /var/www/<project>-uploads  (上传 / 预览文件)
    /etc/letsencrypt           → /etc/letsencrypt            (证书)
```

**核心原则**：
- 网关承担 TLS、路由、静态直发；应用容器以**裸服务**接入。
- 所有项目挂在同一外部网络 `airise-web`，网关用**容器名**反代。
- 静态与上传文件放宿主机路径，网关只读挂载、`sendfile` 直发。
- **conf 自包含**：站点配置在镜像构建期 COPY 进去（非目录挂载），配置变更需 `--build`。

## 3.5 路由策略选型：子域名 vs 子路径

每个新项目接入网关，首先要决定路由方式：**子域名**（`<project>.airise.site`）或**子路径**（`airise.site/<project>/`）。本架构**统一采用子域名**。

### 对比

| 维度 | 子域名 `<project>.airise.site` ✅ | 子路径 `airise.site/<project>/` |
|---|---|---|
| Cookie / localStorage 隔离 | 天然隔离（不同 origin） | ❌ 全共享，项目间互相污染风险 |
| 前端构建 | 零改动，`/` 即根 | ❌ 须配 `base` 前缀，静态资源 + 路由全带前缀 |
| SPA 路由回退 | `try_files ... /index.html` | 须 `try_files ... /<project>/index.html`，history 路由全带前缀 |
| 后端 API 前缀 | `/api/` 不变 | 易与主站 `/api` 冲突，须每项目改前缀 |
| WS / 上传路径 | 各自独立 | 前缀管理成本随项目数线性增长，漏配即踩「signal 卡 pending」 |
| 证书 | 通配证书 `*.airise.site` 一次覆盖 | 单证书即可 |
| DNS | 每项目一条 A，或泛解析 `*` | 一条 |
| 跨项目同源 | ❌ 跨域，需 CORS | ✅ 同源 |
| 日志按项目拆分 | `server_name` 天然分流 | 需按 `$uri` 前缀区分 |

### 选子域名的依据（针对当前场景）

1. **前端零侵入**：子路径要给每个项目的前端 `base`、API 前缀、WS 路径做前缀改造，每加一项目重复一次；子域名无此成本。
2. **origin 隔离**：项目都有登录态 / JWT / 信令池，按 origin 存储。子路径下所有项目挤在同一 origin，token key 重名会互相覆盖。
3. **WS 信令路径**：子路径下 `/api/rtc/signal` 须变 `/<project>/api/rtc/signal`，前端 WS URL、后端路由、nginx location 三处同步改，漏一处即回到信令 pending。
4. **子路径的两个优势（免 DNS、单证书）在此不成立**：DNSPod 泛解析 + 通配证书（DNS-01）后，加项目的边际成本已是 0。

> 唯一考虑切子路径的情形：仅 1 个对外项目且不愿碰 DNS / 证书。本场景多项目，不适用。

## 4. 目录与挂载约定

> 仓库 [`nginx/`](../../nginx/) 目录镜像此结构，含实际配置文件（`conf.d/`、`snippets/`、`README.md`），用 git 同步到服务器 `/opt/nginx/`。

### 网关配置（宿主机）

```
/opt/nginx/
├── conf.d/                         # 多站点（目录挂载进容器）
│   ├── airc.conf                   # ripro（既有，从单文件迁移过来）
│   ├── pixelpack.airise.site.conf  # PixelPack
│   └── _template.conf              # 新项目模板
└── snippets/                       # （可选）可复用片段
    └── proxy-ws.conf
```

### 项目数据（宿主机）

```
/opt/pixelpack/
├── web/dist/        # SPA 构建产物（git pull 后 npm run build 生成）
└── data/
    ├── data.db      # SQLite
    └── uploads/     # 上传文件（api rw 写入，网关 ro 直发）
```

## 5. 关键配置片段

### 5.1 反代通用参数（WS / SSE / 上传）

每个 `proxy_pass` 的 location 都应包含：

```nginx
proxy_http_version 1.1;
map $http_upgrade $connection_upgrade { default upgrade; '' close; }  # 放 http{} 上下文
proxy_set_header Upgrade $http_upgrade;
proxy_set_header Connection $connection_upgrade;
proxy_set_header Host $host;
proxy_set_header X-Real-IP $remote_addr;
proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
proxy_set_header X-Forwarded-Proto $scheme;
proxy_buffering off;          # SSE / WS 必须
proxy_read_timeout 600s;
client_max_body_size 25m;     # 上传上限
```

### 5.2 新项目模板 `conf.d/_template.conf`

```nginx
server {
    listen 443 ssl;
    http2 on;
    server_name <project>.airise.site;

    ssl_certificate     /etc/letsencrypt/live/airise.site/fullchain.pem;   # 通配证书
    ssl_certificate_key /etc/letsencrypt/live/airise.site/privkey.pem;
    ssl_protocols TLSv1.2 TLSv1.3;

    root  /var/www/<project>;
    index index.html;
    client_max_body_size 25m;

    location /api/ {
        proxy_pass http://<project>-api:8000;        # 容器名，需同在 airise-web
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection "upgrade";
        proxy_set_header Host $host;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
        proxy_buffering off;
        proxy_read_timeout 600s;
    }

    location /uploads/ {
        alias /var/www/<project>-uploads/;
        add_header Content-Disposition "inline";     # 预览：浏览器内联
    }

    location / { try_files $uri $uri/ /index.html; }
}

server {
    listen 80;
    server_name <project>.airise.site;
    return 301 https://$host$request_uri;
}
```

### 6.3 网关镜像与容器（仓库 `nginx/`）

网关是**自包含独立容器** `airise-gateway`，conf 在构建期烤进镜像，运行期只挂会变的数据。见仓库 `nginx/Dockerfile` 与 `nginx/docker-compose.yml`：

```dockerfile
# nginx/Dockerfile
FROM nginx:alpine
RUN rm -f /etc/nginx/conf.d/default.conf
COPY conf.d/   /etc/nginx/conf.d/      # 站点配置构建期内置
COPY snippets/ /etc/nginx/snippets/
EXPOSE 80 443
```

```yaml
# nginx/docker-compose.yml
services:
  nginx:
    build: .
    image: airise-gateway
    container_name: airise-gateway
    restart: unless-stopped
    ports: ["80:80", "443:443"]            # 独占宿主机 80/443
    volumes:
      - /etc/letsencrypt:/etc/letsencrypt:ro
      - /opt/pixelpack/web/dist:/var/www/pixelpack:ro
      - /opt/pixelpack/data/uploads:/var/www/pixelpack-uploads:ro
    networks: [airise-web]
networks:
  airise-web: { external: true }
```

> ripro 迁移：删旧 `airc-nginx` 前，把 ripro 的 `airc.conf` 改写为 `conf.d/<ripro>.airise.site.conf`（照抄 `_template.conf`，改 server_name / root / proxy_pass），并在 compose 里追加 ripro 的 dist 与 uploads 挂载，然后 `docker compose up -d --build`。

## 7. 实施步骤

### 7.1 准备数据与网络

```bash
docker network create airise-web

mkdir -p /opt/pixelpack/web /opt/pixelpack/data/uploads
chown -R 1000:1000 /opt/pixelpack/data          # 非 root api 容器用户

# SPA 产物
cd /opt/pixelpack/web && git clone <repo> . && npm install && npm run build
```

### 7.2 同步网关配置并构建启动

```bash
cd /opt/nginx            # 仓库 nginx/ 同步到此处
docker compose up -d --build
docker exec airise-gateway nginx -t
```

### 7.3 部署 PixelPack api

```bash
cd /opt/pixelpack
git pull
cp server/.env.example server/.env && vi server/.env   # 填 GLM token
chown -R 1000:1000 data                                 # 非 root 容器用户权限
docker compose up -d --build
```

## 8. HTTPS 证书

已有 `/etc/letsencrypt`（certbot）。两种方式：

### A. 通配证书 `*.airise.site`（DNS-01，推荐——多项目一劳永逸）

```bash
pip install certbot-dns-dnspod
# /root/dnspod.ini：dns_dnspod_token = <id>,<token>，chmod 600
certbot certonly --dns-dnspod \
  --dns-dnspod-credentials /root/dnspod.ini \
  -d airise.site -d '*.airise.site'
```

所有子域共用 `/etc/letsencrypt/live/airise.site/`，certbot 自动续期。

### B. 单域证书（HTTP-01，免装插件——单项目最快）

```bash
certbot certonly --nginx -d pixelpack.airise.site
# 站点 conf 证书路径改为 /etc/letsencrypt/live/pixelpack.airise.site/
```

> 当前仅接入 PixelPack 一个项目，可先用 B 跑通；加第二个项目时切到 A。

## 9. 新项目接入清单

1. DNS：`<project>.airise.site` A 记录指向服务器（采用子域名路由，理由见 §3.5）。
2. `conf.d/<project>.airise.site.conf`（照抄 `_template.conf`，改 project 名）。
3. 项目 compose：后端容器加入 `airise-web` 外部网络；`/opt/<project>/web/dist` 与 `/opt/<project>/data/uploads` 放到宿主机。
4. 网关 compose 追加两条 bind（dist + uploads），`docker compose up -d --build`（conf 改了也要 `--build`）。
