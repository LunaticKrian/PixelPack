# Nginx 部署架构（多项目网关模式）

> 状态：已上线
> 日期：2026-07-19
> 域名：`airise.site`（腾讯云 / DNSPod）
> 范围：独立网关容器 `airise-gateway`（**独立仓库**），承载多个前后端分离项目（首个接入：PixelPack）。**不与 ripro / airc-nginx 共用容器**。

---

## 1. 背景与目标

服务器上有多个前后端分离项目（PixelPack、ripro-v5-pro、未来更多），都需要：HTTPS / WSS、文件上传下载与预览、SPA 静态托管。早期 PixelPack 采用「内层 nginx（pixelpack-web）+ 外层网关」的**双重 nginx** 结构，内层 nginx 同时处理 SPA 静态、`/api`、`/uploads`、WS 信令，导致 WS/SSE/上传参数要在两层各配一遍，已踩坑（信令 `signal` 卡 pending）。

目标：建立**统一的网关托管模式**，把 TLS、`/api` 反代、`/uploads` 直发、WS/SSE 参数统一收敛到网关一层。前端 SPA 静态资源有两种交付方式（见 §3），**无论哪种，内层服务（若有）必须是纯静态叶子，绝不处理 `/api`、`/uploads`、WS** —— 这才是当年双重 nginx 踩坑的根因，必须避免。

---

## 2. 现状分析

服务器上既有的 `airc-nginx` 对 ripro 项目**已采用规范的网关直发模式**（本架构以此为设计参照，但**新建独立容器 `airise-gateway`，不复用 airc-nginx**）：

| 挂载（ripro） | 用途 |
|---|---|
| `/opt/ripro-v5-pro/frontend/dist` → `/usr/share/nginx/html` (ro) | ripro SPA，网关直发 |
| `ripro-v5-pro_uploads_data` 卷 → `/var/www/uploads` (ro) | ripro 上传文件，网关直发 |
| `/etc/letsencrypt` → `/etc/letsencrypt` (ro) | certbot 证书 |
| `airc.conf` → `conf.d/airc.conf` (ro) | 单站点配置 |

ripro 模式即目标模式。PixelPack 此前的问题：

1. 🔴 **双重 nginx，职责未隔离**：`airc-nginx → pixelpack-web(nginx) → api`，内层 nginx 同时管静态、`/api`、`/uploads`、WS，WS/SSE 配置需双份。
2. 🟡 容器互联用 `172.17.0.1`（docker0 网桥 IP），脆弱。
3. 🟡 静态/上传走后端（FastAPI StaticFiles），不如 nginx 直发高效。
4. 🟡 多站点 conf.d 是单文件挂载，扩展性差。

> 注：PixelPack 现在又引入了一个 `pixelpack-web` 容器（多阶段镜像自带 nginx），但**它是纯静态叶子**（只 `try_files` + gzip + 缓存），`/api`、`/uploads`、WS 全部仍由网关单层处理 —— 与当年「内层 nginx 啥都管」的踩坑结构有本质区别（见 §3 交付方式 B）。

## 3. 目标架构

```
                    Internet (HTTPS / WSS)
                          │
              ┌───────────▼───────────┐
              │  airise-gateway (网关) │  官方 nginx:alpine，纯路由器
              │  · 唯一入口：TLS 终结 + 路由
              │  · /api   → 反代后端容器（WS/SSE 参数统一在此配）
              │  · /     → 反代前端容器（SPA + /uploads 都走这条，PixelPack）
              │  · 通配证书 *.airise.site
              │  · conf.d/ snippets/ 证书 卷挂载，reload 生效
              │  · 不挂任何项目宿主路径（只认容器名）
              └───────────┬───────────┘
                          │ docker network: airise-web (external, 共享)
            ┌─────────────┼─────────────┬──────────────┐
            ▼             ▼             ▼              ▼
      pixelpack-web   pixelpack-api   ripro-backend   appN
      (静态叶子：       (后端裸服务，    (宿主 dist/uploads
       SPA + /uploads   写 ./data/uploads) 由网关直发，方式 A)
       直发；挂 ./data/uploads:ro)
      (容器名直连，不再暴露宿主机端口、不再用 172.17.0.1)

  宿主路径（仅方式 A 项目需要网关挂载；方式 B 项目网关零挂载）：
    方式 A：/opt/<project>/web/dist、/opt/<project>/data/uploads → 网关 ro 直发
    方式 B：网关不挂；SPA 与 uploads 由项目 web 容器从本 compose 的 ./data 自挂
    通用：/etc/letsencrypt → /etc/letsencrypt (证书，网关挂)
```

### 前端 SPA（及 /uploads）的两种交付方式

网关 `location /` 指向前端，有两种等价做法，按项目构建习惯二选一。`/uploads` 跟随 SPA 走同一种方式：

| | 方式 A：网关直发宿主文件 | 方式 B：反代纯静态容器（PixelPack 采用） |
|---|---|---|
| 构建 | 宿主机 `npm run build` → `/opt/<project>/web/dist` | 多阶段镜像（`node` 构建 → `nginx` serve），随 `docker compose up --build` |
| SPA 网关挂载/conf | bind-mount `web/dist`，`root` + `try_files` | 不挂载；`location / { proxy_pass http://<project>-web; }` |
| `/uploads` 谁直发 | 网关：bind-mount `data/uploads`，`location /uploads/ { alias ...; }` | web 容器：挂 `./data/uploads:ro`，其 nginx `location /uploads/ { alias ...; }`，网关 `/uploads` 随 `location /` 转发 |
| 路径漂移风险 | 有：网关与项目是两个 compose，硬编码绝对路径易漂移 | 无：api 写、web 读都在同一 compose 用 `./data`，宿主路径天然一致 |
| 更新前端 | `git pull && npm run build`，reload 即生效 | `docker compose up -d --build web`（重建镜像） |
| 适用 | 宿主机有 node、想免重建镜像（ripro 模式） | 前端全容器化、不依赖宿主 node（PixelPack 模式） |

> **关键约束（两种方式共同）**：内层服务（方式 B 的 `<project>-web` 容器）**只处理静态资源**（SPA + `/uploads`），`/api`、WS 信令一律由网关单层直连后端，不进内层。`/uploads` 虽经 web，但它是静态文件直发（alias），不涉及 WS/SSE/上传写入参数，故不违反此约束。这才能避免当年双重 nginx 的「WS/SSE/上传参数双份配置」踩坑。

**核心原则**：
- 网关是**纯路由器**：承担 TLS + `/api` 反代（+ WS/SSE 参数）+ 其余转发到前端；**不挂任何项目宿主路径**（方式 B），只按容器名寻址。
- 所有项目挂在同一外部网络 `airise-web`，网关用**容器名**反代。
- 上传文件：方式 A 由网关直发宿主路径；方式 B 由项目 web 容器直发（api 写、web 读同属一个 compose，零漂移）。
- **conf 卷挂载**：站点配置（`conf.d/`）只读挂载进容器，变更后 `nginx -s reload` 即生效，无需重建镜像。

## 3.5 路由策略选型：子域名 vs 子路径

每个新项目接入网关，首先要决定路由方式：**子域名**（`<project>.airise.site`）或**子路径**（`airise.site/<project>/`）。本架构**统一采用子域名**。

### 对比

| 维度 | 子域名 `<project>.airise.site` ✅ | 子路径 `airise.site/<project>/` |
|---|---|---|
| Cookie / localStorage 隔离 | 天然隔离（不同 origin） | ❌ 全共享，项目间互相污染风险 |
| 前端构建 | 零改动，`/` 即根 | ❌ 须配 `base` 前缀，静态资源 + 路由全带前缀 |
| SPA 路由回退 | `try_files ... /index.html` | 须 `try_files ... /<project>/index.html`，history 路由全带前缀 |
| 后端 API 前缀 | `/api/` 不变 | 易与主站 `/api` 冲突，须每项目改前缀 |
| WS / 上传路径 | 各自独立 | 前缀管理成本随项目数线性增长，漏配即踩「signal 卡 pending」 |
| 证书 | 通配证书 `*.airise.site` 一次覆盖 | 单证书即可 |
| DNS | 每项目一条 A，或泛解析 `*` | 一条 |
| 跨项目同源 | ❌ 跨域，需 CORS | ✅ 同源 |
| 日志按项目拆分 | `server_name` 天然分流 | 需按 `$uri` 前缀区分 |

### 选子域名的依据（针对当前场景）

1. **前端零侵入**：子路径要给每个项目的前端 `base`、API 前缀、WS 路径做前缀改造，每加一项目重复一次；子域名无此成本。
2. **origin 隔离**：项目都有登录态 / JWT / 信令池，按 origin 存储。子路径下所有项目挤在同一 origin，token key 重名会互相覆盖。
3. **WS 信令路径**：子路径下 `/api/rtc/signal` 须变 `/<project>/api/rtc/signal`，前端 WS URL、后端路由、nginx location 三处同步改，漏一处即回到信令 pending。
4. **子路径的两个优势（免 DNS、单证书）在此不成立**：DNSPod 泛解析 + 通配证书（DNS-01）后，加项目的边际成本已是 0。

> 唯一考虑切子路径的情形：仅 1 个对外项目且不愿碰 DNS / 证书。本场景多项目，不适用。

## 4. 目录与挂载约定

> 网关配置在**独立仓库** `airise-gateway`，含 `conf.d/`、`snippets/`、`docker-compose.yml`、`README.md`，用 git 同步到服务器 `/opt/airise-gateway/`。

### 网关配置（宿主机）

```
/opt/airise-gateway/
├── docker-compose.yml              # 独立容器 airise-gateway，绑 80/443，卷挂载
├── conf.d/                         # 多站点（目录挂载进容器）
│   ├── 00-upgrade-map.conf         # 全局 WebSocket 升级映射
│   ├── pixelpack.airise.site.conf  # PixelPack（/ 反代到 pixelpack-web）
│   ├── model.airise.site.conf      # PixelPack 孪生域名
│   └── _template.example           # 新项目模板（非 .conf，nginx 不加载）
└── snippets/
    └── proxy.conf                  # 反代通用参数（WS/SSE/上传/超时）
```

### 项目数据（宿主机）

```
/opt/pixelpack/
├── data/
│   ├── data.db      # SQLite
│   └── uploads/     # 上传文件（api rw 写入 /app/data/uploads，web ro 挂载直发）
└── （前端 dist 不放宿主机：走交付方式 B，由 web 容器多阶段镜像自带）
```

> PixelPack 走交付方式 B：`api` 与 `web` 同属一个 compose，都引用 `./data`（相对路径），宿主绝对路径必然一致 → 上传文件零漂移。网关不挂载此目录。
> 若某项目改用交付方式 A（网关直发宿主 dist + uploads），则需 `/opt/<project>/web/dist` 与 `/opt/<project>/data/uploads`，并在网关 compose 追加这两条卷挂载。

## 5. 关键配置片段

### 5.1 反代通用参数（WS / SSE / 上传）

每个 `proxy_pass` 的 location 都应包含：

```nginx
proxy_http_version 1.1;
map $http_upgrade $connection_upgrade { default upgrade; '' close; }  # 放 http{} 上下文
proxy_set_header Upgrade $http_upgrade;
proxy_set_header Connection $connection_upgrade;
proxy_set_header Host $host;
proxy_set_header X-Real-IP $remote_addr;
proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
proxy_set_header X-Forwarded-Proto $scheme;
proxy_buffering off;          # SSE / WS 必须
proxy_read_timeout 600s;
client_max_body_size 25m;     # 上传上限
```

### 5.2 新项目模板 `conf.d/_template.example`

> 模板默认采用**交付方式 A**（网关直发宿主 dist + uploads）。若新项目走**交付方式 B**（前端容器直发 SPA + uploads），把 `location /` 改为：
> ```nginx
> location / {
>     proxy_pass http://<project>-web;
>     proxy_set_header Host $host;
>     proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
> }
> ```
> 并删去 `root` / `index` 与整个 `location /uploads/` 块（`/uploads` 随 `location /` 转发给 web 容器，由其 nginx 直发）。两种方式的取舍见 §3。

```nginx
server {
    listen 443 ssl;
    http2 on;
    server_name <project>.airise.site;

    ssl_certificate     /etc/letsencrypt/live/airise.site/fullchain.pem;   # 通配证书
    ssl_certificate_key /etc/letsencrypt/live/airise.site/privkey.pem;
    ssl_protocols TLSv1.2 TLSv1.3;

    root  /var/www/<project>;
    index index.html;
    client_max_body_size 25m;

    location /api/ {
        proxy_pass http://<project>-api:8000;        # 容器名，需同在 airise-web
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection "upgrade";
        proxy_set_header Host $host;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
        proxy_buffering off;
        proxy_read_timeout 600s;
    }

    location /uploads/ {
        alias /var/www/<project>-uploads/;
        add_header Content-Disposition "inline";     # 预览：浏览器内联
    }

    location / { try_files $uri $uri/ /index.html; }
}

server {
    listen 80;
    server_name <project>.airise.site;
    return 301 https://$host$request_uri;
}
```

## 6. PixelPack 落地

### 6.1 应用 compose（PixelPack 仓库 `docker-compose.yml`）

`api`（FastAPI/uvicorn）+ `web`（多阶段 nginx 静态，交付方式 B）两个服务，均加入 `airise-web` 外部网络。`api` 写 `./data/uploads`（`UPLOAD_DIR=/app/data/uploads`），`web` 只读挂载 `./data/uploads` 直发 —— 同一 compose、同一 `./data`，宿主路径天然一致，**零漂移**。`web` 不暴露端口，由网关反代。

### 6.2 网关站点 `conf.d/pixelpack.airise.site.conf`

```nginx
server {
    listen 443 ssl;
    http2 on;
    server_name pixelpack.airise.site;

    ssl_certificate     /etc/letsencrypt/live/airise.site/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/airise.site/privkey.pem;
    ssl_protocols TLSv1.2 TLSv1.3;

    client_max_body_size 25m;        # 作用于 /api 的上传写入

    location /api/ {
        proxy_pass http://pixelpack-api:8000;
        include /etc/nginx/snippets/proxy.conf;   # WS/SSE/头/超时 统一在此
    }

    # SPA + /uploads 都转发给前端容器（web-nginx 内部按路径区分）
    location / {
        proxy_pass http://pixelpack-web;
        proxy_set_header Host $host;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    }
}

server {
    listen 80;
    server_name pixelpack.airise.site;
    return 301 https://$host$request_uri;
}
```

### 6.3 网关容器（独立仓库 `airise-gateway`）+ 前端容器 nginx

**网关**是独立容器 `airise-gateway`，官方 `nginx:alpine`，**只卷挂载 conf / snippets / 证书**（不挂任何项目宿主路径），conf 改完 `nginx -s reload` 即生效：

```yaml
# airise-gateway/docker-compose.yml
services:
  nginx:
    image: nginx:alpine
    container_name: airise-gateway
    restart: unless-stopped
    ports: ["80:80", "443:443"]            # 独占宿主机 80/443
    volumes:
      - ./conf.d:/etc/nginx/conf.d:ro      # 站点配置（reload 生效）
      - ./snippets:/etc/nginx/snippets:ro
      - /etc/letsencrypt:/etc/letsencrypt:ro
      # 不挂任何项目路径：SPA、/uploads 全部由 pixelpack-web 提供
    tmpfs: ["/run", "/var/cache/nginx"]    # nginx 运行期 pid/temp
    networks: [airise-web]
networks:
  airise-web: { external: true }
```

**前端容器**（`pixelpack-web`）的 nginx（`web/nginx.conf`，镜像自带）同时 serve SPA 与 `/uploads`：

```nginx
# web/nginx.conf（节选）
root /usr/share/nginx/html;
location /assets/ { expires 1y; add_header Cache-Control "public, immutable"; }
location /uploads/ {                          # ← 直发上传文件
    alias /app/data/uploads/;                 # web 容器挂载 ./data/uploads:ro
    add_header Content-Disposition "inline";
}
location / { try_files $uri $uri/ /index.html; }   # SPA 回退
```

`pixelpack-web` 在 PixelPack compose 里挂载 `./data/uploads:/app/data/uploads:ro`，与 `api` 的 `./data:/app/data` 同源 —— `api` 写入的图片 `web` 立即可读，零漂移。

> ripro 迁移：删旧 `airc-nginx` 前，把 ripro 的 `airc.conf` 改写为 `conf.d/<ripro>.airise.site.conf`（照抄 `_template.example`，改 server_name / root / proxy_pass）。ripro 若用交付方式 A（网关直发），还需在网关 compose 追加 ripro 的 dist + uploads 挂载；用方式 B 则不用。然后 `docker compose up -d` + `docker exec airise-gateway nginx -s reload`。

## 8. HTTPS 证书

> 完整签发与续期步骤（DNSPod DNS-01、续期 hook、单域→通配迁移、故障速查）见
> [`260719-通配证书签发.md`](260719-通配证书签发.md)。本节仅给结论速览。

已有 `/etc/letsencrypt`（certbot）。两种方式：

### A. 通配证书 `*.airise.site`（DNS-01，推荐——多项目一劳永逸）

```bash
pip install certbot-dns-dnspod
# /root/dnspod.ini：dns_dnspod_token = <id>,<token>，chmod 600
certbot certonly --dns-dnspod \
  --dns-dnspod-credentials /root/dnspod.ini \
  -d airise.site -d '*.airise.site'
```

所有子域共用 `/etc/letsencrypt/live/airise.site/`，certbot 自动续期。

### B. 单域证书（HTTP-01，免装插件——单项目最快）

```bash
certbot certonly --nginx -d pixelpack.airise.site
# 站点 conf 证书路径改为 /etc/letsencrypt/live/pixelpack.airise.site/
```

> 当前仅接入 PixelPack 一个项目，可先用 B 跑通；加第二个项目时切到 A。

## 9. 新项目接入清单

1. DNS：`<project>.airise.site` A 记录指向服务器（采用子域名路由，理由见 §3.5）。
2. `conf.d/<project>.airise.site.conf`（照抄 `_template.example`，改 project 名；`location /` 与 `/uploads` 按交付方式 A/B 二选一）。
3. 项目 compose：后端容器加入 `airise-web` 外部网络；上传目录放宿主机 `./data/uploads`（api 写）。
   - 交付方式 A：宿主 `npm run build` 出 dist，网关 compose 追加 dist + uploads 卷挂载，网关 `root`/`alias` 直发。
   - 交付方式 B：项目 compose 自带 `<project>-web` 多阶段容器（挂 `./data/uploads:ro`、nginx 内 `location /uploads/` 直发），加入 `airise-web`；网关 `location /` 反代之，**网关零挂载**。
4. 网关侧改动：方式 A 需 `docker compose up -d`（新挂载卷）；方式 B 只改 conf → `docker exec airise-gateway nginx -s reload`（卷挂载，无需 `--build`）。

## 10. 历史变更（前端容器的去而复返 + uploads 归属）

PixelPack 的前端容器经历过两次演变，记录在此避免再踩：

- **初版（已废弃）**：`pixelpack-web` 是个**啥都管**的内层 nginx（静态 + `/api` + `/uploads` + WS），与外层网关形成双重 nginx，WS/SSE 参数双份配置，踩坑 `signal` 卡 pending。该结构连同 `web/Dockerfile`、`web/nginx.conf`、compose 的 `web` 服务一度被整体删除，改为「网关直发宿主 dist」。
- **二版**：重新引入 `web` 容器（`web/Dockerfile` 多阶段：`node` 构建 → `nginx` serve），限定为纯静态叶子（只 SPA），`/api`、`/uploads`、WS 仍在网关单层。但 `/uploads` 由网关挂载宿主路径直发 —— 网关与 PixelPack 是两个 compose，硬编码绝对路径，**换部署目录即路径漂移、图片 404**（线上踩过）。
- **现行版（三版）**：把 `/uploads` 也交给 `web` 容器直发（web-nginx 加 `location /uploads/` alias，web 挂 `./data/uploads:ro`）。网关瘦身成**纯路由器**，不挂任何项目宿主路径；`api` 写、`web` 读同属一个 compose 的 `./data`，**零漂移**。`/uploads` 虽经 web，但纯静态直发，不涉及 WS/SSE 参数，不违反「内层不碰 /api/WS」约束。详见 §3 交付方式 B。
