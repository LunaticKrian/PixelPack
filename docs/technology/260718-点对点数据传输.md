# 点对点（P2P）数据传输技术方案

> 状态：设计稿
> 日期：2026-07-18
> 范围：在现有「浏览器 → FastAPI 服务端」的 CS 架构上，新增**两个登录用户之间的局域网（LAN）点对点文件传输**能力。

---

## 1. 背景与目标

当前 PixelPack 为标准 CS 架构：浏览器（Vue） ↔ 服务端（FastAPI）。所有数据都经由服务器中转，既消耗服务器带宽，也无法做到「同在一个房间里、却走直连」的高效传输。

目标：让**两个已登录用户**在浏览器里直接互传文件，数据流**不经过服务端**，服务端仅承担「撮合双方建立连接」的轻量职责。

### 选型结论

| 维度 | 选择 | 理由 |
|---|---|---|
| 传输技术 | **WebRTC DataChannel** | 浏览器沙箱内**无法开启原生 TCP/UDP socket**；WebRTC 是浏览器唯一的 P2P 通道，自带 DTLS 加密、可传任意二进制、可流式。 |
| 服务端角色 | **信令服务器（WebSocket）** | 仅交换 SDP / ICE，不承载数据流，负载极小。 |
| 传输内容 | **任意文件** | DataChannel 分块传输；后续可在此通道上扩展业务数据。 |
| 配对方式 | **在线用户列表 + 邀请码** | 在线列表让发起方知道对方在不在；邀请码保证接收方**主动同意**，避免被陌生人随意打扰。 |

---

## 2. 总体架构

```
浏览器A ──┐                       ┌── 浏览器B
          │  HTTPS / WSS（信令）   │
          └──> FastAPI 信令 <─────┘
                    │
            交换 SDP / ICE candidate 完成：
   浏览器A ════ WebRTC DataChannel ════ 浏览器B
                 （数据直连，不过服务器）
```

- **信令阶段**：两端各持一个到 `/api/rtc/signal` 的 WebSocket（带 JWT 鉴权）。服务端维护「在线用户池」与「邀请码 → 待连接 offer」映射，在两端间转发 `offer / answer / candidate`。
- **数据阶段**：连接建立后，文件经 DataChannel 直接在两浏览器之间传输，服务端不参与。

### 与现有 CS 架构的关系

- 服务端**职责不变**，只多挂一个 WebSocket 端点，不承担文件流量。
- 鉴权**复用现有 JWT**（`app/utils/security.py::verify_token` + `app/utils/deps.py`），不另起一套。
- WebRTC 通道是「传输层」；传完文件若要落库，由前端再调现有 `/api/...` 写库即可，与 P2P 解耦。

## 3. 关键约束与前置条件

### 3.1 必须 HTTPS（安全上下文）

浏览器的 `RTCPeerConnection` / DataChannel **只在安全上下文下可用**（HTTPS 或 `localhost`）。在 `http://` 非 localhost 站点上，WebRTC API 会被浏览器**禁用**。

> 本项目线上已通过 Nginx 提供 HTTPS，满足此条件。开发期用 `localhost` 也可。
>
> 连带要求：信令 WebSocket 必须用 `wss://`（HTTPS 页面不允许连接 `ws://`）。

### 3.2 Nginx 必须支持 WebSocket 升级

现有 `/api` 反代为 SSE 配置（关 buffering）。WebRTC 信令走 WebSocket，需额外补 Upgrade 头：

```nginx
location /api/ {
    proxy_pass http://api:8000;
    proxy_http_version 1.1;
    proxy_set_header Upgrade $http_upgrade;          # ← 新增
    proxy_set_header Connection "upgrade";            # ← 新增
    proxy_set_header Host $host;
    # SSE / WebSocket 都需要
    proxy_buffering off;
    proxy_cache off;
    proxy_read_timeout 600s;
}
```

### 3.3 局域网直连 vs 跨网中继

| 场景 | 连接方式 |
|---|---|
| 两端**同一局域网** | host candidate 直接连通，**纯 P2P**，数据不出 LAN ✅ |
| 不同子网 / 移动网络 | 需 STUN 打洞；打不通则需 **TURN 中继**（此时退化为经服务器转发，非纯 P2P） |

本方案默认 `iceServers: []`（局域网直连优先）。如未来需跨网，再追加 STUN/TURN 配置。

### 3.4 Wi-Fi AP 隔离

很多路由器默认开启「客户端隔离」，会拦截设备间直连。部署前让两端互 `ping` 确认网络层可达。

## 4. 详细流程

### 4.1 在线与信令

1. 用户登录后打开传输页，前端用现有 access token 建立 WebSocket：
   `wss://host/api/rtc/signal?token=<JWT>`
2. 服务端校验 JWT（复用 `verify_token`），成功后把 `user_id` 加入在线池，并向所有在线用户广播当前在线列表（含 `user_id` / 用户名）。
3. 断开时从池中移除并重广播。

### 4.2 邀请码配对（同意制）

**发起方 A：**

1. A 在前端 `new RTCPeerConnection({ iceServers: [] })`，创建 DataChannel，生成 `offer`。
2. A 经 WS 发 `{ type: "invite_create", offer }` 给服务端。
3. 服务端生成 6 位短码 `code`，存 `code → { from: A, offer }`，返回给 A。A 页面展示该码。

**接收方 B：**

1. B 在输入框填入 `code`，经 WS 发 `{ type: "invite_join", code }`。
2. 服务端查到对应记录，把 A 的 `offer` 下发给 B。
3. B `setRemoteDescription(offer)` → `createAnswer()` → 经 WS 回传 `{ type: "answer", to: A, answer }`。
4. 服务端转发 answer 给 A；A `setRemoteDescription`。
5. 双方互发 ICE candidate（`{ type: "candidate", to, candidate }`），服务端按 `to` 转发。
6. DataChannel `onopen` 触发 → 进入数据阶段。

> 「在线列表 + 邀请码」组合的好处：A 从列表看到 B 在线后发起，B 输码即表示**同意接收**，杜绝被随意连接。码使用一次即失效。

### 4.3 文件分块传输协议

DataChannel 单条消息建议 ≤ 16KB（更稳），且必须等接收方回执做流控，避免缓冲爆满。约定一个简单应用层协议：

```
# 起始
{ kind: "meta", name, size, mime }
# 数据块（ArrayBuffer，offset 用于拼装与进度）
{ kind: "chunk", offset } + 二进制
# 接收方收到一块后回 ACK（流量控制 / 滑窗）
{ kind: "ack", offset }
# 结束
{ kind: "done" }
# 接收方校验完整性
{ kind: "ok" | "fail" }
```

- 发送方维护一个**滑动窗口**（如 4 块在途），收到 ack 再发下一批，兼顾吞吐与背压。
- 接收方按 `offset` 顺序写入 `Blob`（或 `FileSystemWritableFileStream` 落地大文件），收齐后触发下载。
- 进度条：前端按 `offset / size` 计算。

## 5. 实现拆解

### 5.1 后端：信令路由 `server/app/routers/rtc.py`

职责：WS 鉴权、在线池、邀请码撮合、SDP/ICE 转发。约 60–80 行。

核心结构：

```python
from fastapi import APIRouter, WebSocket, WebSocketDisconnect
from jose import JWTError

from app.utils.security import verify_token

router = APIRouter(prefix="/api/rtc")

# user_id -> WebSocket
online: dict[int, WebSocket] = {}
# 邀请码 -> {"from": user_id, "offer": sdp}
invites: dict[str, dict] = {}


async def _broadcast_online():
    payload = {"type": "presence", "users": [{"id": uid} for uid in online]}
    for ws in list(online.values()):
        await ws.send_json(payload)


@router.websocket("/signal")
async def signal(ws: WebSocket, token: str):
    # 1) 鉴权（复用现有 JWT）
    try:
        payload = verify_token(token)
        assert payload.get("type") == "access"
        uid = int(payload["sub"])
    except (JWTError, AssertionError, KeyError):
        await ws.close(code=4401)
        return

    await ws.accept()
    online[uid] = ws
    await _broadcast_online()

    try:
        while True:
            msg = await ws.receive_json()
            t = msg.get("type")

            if t == "invite_create":
                code = _gen_code()                       # 6 位短码
                invites[code] = {"from": uid, "offer": msg["offer"]}
                await ws.send_json({"type": "invite_created", "code": code})

            elif t == "invite_join":
                rec = invites.pop(msg["code"], None)
                if rec:                                  # 把发起方 offer 下发
                    await ws.send_json({"type": "offer",
                                        "from": rec["from"],
                                        "offer": rec["offer"]})
                else:
                    await ws.send_json({"type": "invite_not_found"})

            elif t in ("answer", "candidate"):
                target = online.get(int(msg["to"]))
                if target:
                    await target.send_json(
                        {"type": t, "from": uid,
                         **({"answer": msg["answer"]} if t == "answer"
                            else {"candidate": msg["candidate"]})})
    except WebSocketDisconnect:
        pass
    finally:
        online.pop(uid, None)
        await _broadcast_online()
```

> 注：在线用户名展示可在广播时按需 join 用户表，或前端复用已有用户信息接口。

在 `app/main.py` 注册：`app.include_router(rtc.router)`。

### 5.2 前端：传输页 `web/src/views/Transfer.vue`

职责：建立信令 WS、展示在线列表、生成 / 输入邀请码、发起 WebRTC、分块收发文件。

骨架（仅核心流程）：

```ts
// 信令（token 来自 auth store）
const ws = new WebSocket(`wss://${location.host}/api/rtc/signal?token=${token}`)

const pc = new RTCPeerConnection({ iceServers: [] })
let dc: RTCDataChannel

pc.onicecandidate = (e) =>
  e.candidate && ws.send(JSON.stringify(
    { type: 'candidate', to: targetId, candidate: e.candidate }))

ws.onmessage = async (e) => {
  const m = JSON.parse(e.data)
  switch (m.type) {
    case 'presence':       onlineUsers.value = m.users; break
    case 'offer':          // 我是接收方
      pc.ondatachannel = (ev) => (dc = ev.channel)
      await pc.setRemoteDescription(m.offer)
      const ans = await pc.createAnswer()
      await pc.setLocalDescription(ans)
      ws.send(JSON.stringify({ type: 'answer', to: m.from, answer: ans }))
      break
    case 'answer':         // 我是发起方
      await pc.setRemoteDescription(m.answer); break
    case 'candidate':
      await pc.addIceCandidate(m.candidate); break
  }
}

// 发起：创建 DataChannel + offer，拿邀请码
async function createInvite() {
  dc = pc.createDataChannel('file')
  const offer = await pc.createOffer()
  await pc.setLocalDescription(offer)
  ws.send(JSON.stringify({ type: 'invite_create', offer }))
}
// 收到 invite_created 后展示 code

// 接收方填码
ws.send(JSON.stringify({ type: 'invite_join', code }))
```

**文件收发**（分块发送、按 offset 拼装接收）：

```ts
const CHUNK = 16 * 1024

async function sendFile(file: File) {
  dc.send(JSON.stringify({ kind: 'meta', name: file.name, size: file.size, mime: file.type }))
  for (let offset = 0; offset < file.size; offset += CHUNK) {
    const buf = await file.slice(offset, offset + CHUNK).arrayBuffer()
    dc.send(buf)                 // 生产实现需加 ack 滑窗 + 背压
  }
  dc.send(JSON.stringify({ kind: 'done' }))
}

dc.onmessage = (e) => {
  // meta：建 Blob[]；chunk：push 并存 offset；done：合成 Blob 触发下载
}
```

> 生产实现需补：滑窗 + ack、错误重传策略、大文件用 `showSaveFilePicker` 流式落盘。

## 6. 安全考虑

| 风险 | 应对 |
|---|---|
| 陌生人接信令 | WS 握手校验 JWT（失败以 4401 关闭）；不在线、未认证一律拒绝。 |
| 被随意发起连接 | 邀请码一次性 + 接收方主动输入，等于显式同意。 |
| 明文传输 | WebRTC DataChannel 默认 DTLS 加密，端到端。 |
| 大文件占用浏览器内存 | 分块 + 滑窗 + 流式落盘（`FileSystemWritableFileStream`），避免整文件入内存。 |
| 邀请码爆破 | 6 位码、短时效（如 60s 过期）、尝试次数限制（可选）。 |

---

## 7. 限制与后续演进

- **跨网传输**：当前仅保证局域网纯 P2P；跨网需引入 STUN/TURN。可自托管 coturn 或用公共 STUN（如 `stun:stun.l.google.com:19302`）。
- **NAT 类型**：对称型 NAT 基本打不通，必须 TURN 兜底。
- **多对多 / 群传**：当前一对一；后续可扩展为多方 Mesh 或 SFU（超出本方案范围）。
- **断点续传**：基于 offset 协议可实现，需持久化传输进度。
- **移动端后台**：手机锁屏 / 切后台后 WebSocket 与 DataChannel 易被系统挂起，需提示用户保持页面在前台。
