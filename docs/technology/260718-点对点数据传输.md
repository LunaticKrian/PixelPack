# 点对点（P2P）数据传输技术方案

> 状态：设计稿
> 日期：2026-07-18
> 范围：在现有「浏览器 → FastAPI 服务端」的 CS 架构上，新增**两个登录用户之间的局域网（LAN）点对点文件传输**能力。

---

## 1. 背景与目标

当前 PixelPack 为标准 CS 架构：浏览器（Vue） ↔ 服务端（FastAPI）。所有数据都经由服务器中转，既消耗服务器带宽，也无法做到「同在一个房间里、却走直连」的高效传输。

目标：让**两个已登录用户**在浏览器里直接互传文件，数据流**不经过服务端**，服务端仅承担「撮合双方建立连接」的轻量职责。

### 选型结论

| 维度 | 选择 | 理由 |
|---|---|---|
| 传输技术 | **WebRTC DataChannel** | 浏览器沙箱内**无法开启原生 TCP/UDP socket**；WebRTC 是浏览器唯一的 P2P 通道，自带 DTLS 加密、可传任意二进制、可流式。 |
| 服务端角色 | **信令服务器（WebSocket）** | 仅交换 SDP / ICE，不承载数据流，负载极小。 |
| 传输内容 | **任意文件** | DataChannel 分块传输；后续可在此通道上扩展业务数据。 |
| 配对方式 | **在线用户列表 + 邀请码** | 在线列表让发起方知道对方在不在；邀请码保证接收方**主动同意**，避免被陌生人随意打扰。 |

---

## 2. 总体架构

```
浏览器A ──┐                       ┌── 浏览器B
          │  HTTPS / WSS（信令）   │
          └──> FastAPI 信令 <─────┘
                    │
            交换 SDP（offer/answer，各自裹好候选）完成：
   浏览器A ════ WebRTC DataChannel ════ 浏览器B
                 （数据直连，不过服务器）
```

- **信令阶段**：两端各持一个到 `/api/rtc/signal` 的 WebSocket（首帧 JWT 鉴权）。服务端维护「在线用户池」与「邀请码 → 待连接 offer」映射，在两端间转发 `offer / answer`（候选已裹在 SDP 内，**不单独转发 candidate**）。
- **数据阶段**：连接建立后，文件经 DataChannel 直接在两浏览器之间传输，服务端不参与。

### 与现有 CS 架构的关系

- 服务端**职责不变**，只多挂一个 WebSocket 端点，不承担文件流量。
- 鉴权**复用现有 JWT**（`app/utils/security.py::verify_token` + `app/utils/deps.py`），不另起一套。
- WebRTC 通道是「传输层」；传完文件若要落库，由前端再调现有 `/api/...` 写库即可，与 P2P 解耦。

## 3. 关键约束与前置条件

### 3.1 必须 HTTPS（安全上下文）

浏览器的 `RTCPeerConnection` / DataChannel **只在安全上下文下可用**（HTTPS 或 `localhost`）。在 `http://` 非 localhost 站点上，WebRTC API 会被浏览器**禁用**。

> 本项目线上已通过 Nginx 提供 HTTPS，满足此条件。开发期用 `localhost` 也可。
>
> 连带要求：信令 WebSocket 必须用 `wss://`（HTTPS 页面不允许连接 `ws://`）。

### 3.2 Nginx 必须支持 WebSocket 升级

现有 `/api` 反代为 SSE 配置（关 buffering）。WebRTC 信令走 WebSocket，需额外补 Upgrade 头：

```nginx
location /api/ {
    proxy_pass http://api:8000;
    proxy_http_version 1.1;
    proxy_set_header Upgrade $http_upgrade;          # ← 新增
    proxy_set_header Connection "upgrade";            # ← 新增
    proxy_set_header Host $host;
    # SSE / WebSocket 都需要
    proxy_buffering off;
    proxy_cache off;
    proxy_read_timeout 600s;
}
```

### 3.3 局域网直连 vs 跨网中继

| 场景 | 连接方式 |
|---|---|
| 两端**同一局域网** | host candidate 直接连通，**纯 P2P**，数据不出 LAN ✅ |
| 不同子网 / 移动网络 | 需 STUN 打洞；打不通则需 **TURN 中继**（此时退化为经服务器转发，非纯 P2P） |

本方案默认 `iceServers: []`（局域网直连优先）。如未来需跨网，再追加 STUN/TURN 配置。

### 3.4 Wi-Fi AP 隔离

很多路由器默认开启「客户端隔离」，会拦截设备间直连。部署前让两端互 `ping` 确认网络层可达。

### 3.5 ICE 与 trickle / 非 trickle 背景

> 本节是 §4.2「非 trickle 决策」的前置知识。

WebRTC 要在两个浏览器间建直连，需先找出「能用哪些网络地址连通对方」，每找出一个可能的地址就叫一个 **ICE 候选（candidate）**。主要有三类：

| 类型 | 含义 | 例子 |
|---|---|---|
| host | 本机网卡地址（局域网 IP） | `192.168.1.5:60000` |
| srflx | 经 STUN 查到的公网映射地址 | `203.0.113.7:51234` |
| relay | TURN 中继地址（打不通时的兜底） | TURN 服务器地址 |

收集候选需要时间——host 候选瞬间就有，但 srflx 要往 STUN 往返一次，relay 更慢。收集到一批后，两端把候选两两配对、做连通性测试（ICE check），找到第一对能通的即建立连接。核心问题在于：**候选何时、如何交给对方？** 这就是 trickle 与非 trickle 的区别。

**非 trickle（传统 / vanilla）**——等全部候选收集完，一次性塞进 SDP（offer/answer）发出去。

```
A: 收集候选…(等 STUN 返回)…收集完 → 全部写进 offer → 发 offer
B: 收到完整 offer → 收集自己的…完 → 写进 answer → 发 answer
→ 两端各有对方全部候选 → 配对测试 → 连上
```

- ✅ 信令最简单：一次 offer、一次 answer，各自自带全部候选，无额外消息类型。
- ❌ 握手慢：必须等候选收完才能发 SDP。

**trickle（流式 / 增量）**——SDP 立刻发（不含/少含候选），候选收集到一个就单独发一个，像滴水一样"滴"过去。

```
A: 立刻发 offer（空候选）
A: host 好了 → 发 {candidate1}
A: srflx 好了 → 发 {candidate2}     ← 边收集边发，对方边收边测
B: 收到 offer → 立刻发 answer（空候选）
B: 候选好了 → 发 {candidate3}
→ 两端各自 addIceCandidate()，凑齐第一对能通的就连上
```

- ✅ 连接更快：不用等收集完，第一个候选到了就能开始测试，收集与测试并行。
- ❌ 信令更复杂：要处理很多条候选消息的转发、乱序，以及「远端 SDP 还没 setRemoteDescription 时收到候选要先缓存」等时序问题。

| | 非 trickle | trickle |
|---|---|---|
| 候选怎么传 | 全裹进 SDP，一次发 | 单独消息，一个个滴过去 |
| 握手速度 | 慢（等收集完） | 快（收集与测试并行） |
| 信令复杂度 | 极简 | 复杂（转发/缓存/乱序） |
| 典型场景 | 局域网、信令受限、简单系统 | 公网跨 NAT、追求快速建连 |

本方案选**非 trickle**，原因见 §4.2。

## 4. 详细流程

### 4.1 在线与信令

1. 用户登录后打开传输页，前端建立 WebSocket `wss://host/api/rtc/signal`，
   **建连后第一帧发 `{ token }`** 做鉴权（token 不走 URL，避免进入 nginx access log / 浏览器历史）。
2. 服务端用 `verify_token` 校验首帧 JWT（复用现有鉴权），成功后解析用户、把 `user_id` 加入在线池，并向所有在线用户广播当前在线列表（含 `user_id` / 用户名）。失败则发 `auth_failed` 并以 4401 关闭。
3. 断开时从池中移除并重广播。

### 4.2 邀请码配对（同意制）

**发起方 A：**

1. A 在前端 `new RTCPeerConnection({ iceServers: [] })`，创建 DataChannel，`createOffer()` + `setLocalDescription(offer)`。
2. A **等待 ICE 收集完成**（`iceGatheringState === 'complete'`，或带超时兜底），此时 `pc.localDescription` 已包含全部候选。
3. A 经 WS 发 `{ type: "invite_create", offer: pc.localDescription }`（offer 自带候选）。
4. 服务端生成 6 位短码 `code`，存 `code → { from: A, offer }`，返回给 A。A 页面展示该码。

**接收方 B：**

1. B 在输入框填入 `code`，经 WS 发 `{ type: "invite_join", code }`。
2. 服务端查到对应记录，把 A 的 `offer` 下发给 B（offer 已含 A 的全部候选）。
3. B `setRemoteDescription(offer)` → `createAnswer()` + `setLocalDescription(answer)` → 同样**等 ICE 收集完成**，再把 `pc.localDescription`（answer 自带候选）经 WS 回传 `{ type: "answer", to: A, answer }`。
4. 服务端把 answer 转给 A；A `setRemoteDescription`。两端 SDP 互换完毕，连接建立。
5. DataChannel `onopen` 触发 → 进入数据阶段。

> **实现决策：非 trickle ICE。** offer / answer 各自等 ICE 收集完成后把候选裹进 SDP 一次发完，因此**无需任何 candidate 中继**。原因是「先发邀请码、后配对」场景下，A 在 B 加入前就会产生候选，而 A 此时不知道 B 的 user_id，trickle 候选无法寻址 → 早期候选会被丢弃、连接随机失败。非 trickle 以「生成码前多等 1~3 秒」换取信令的简化与稳定，对局域网场景完全可接受。

> 「在线列表 + 邀请码」组合的好处：A 从列表看到 B 在线后发起，B 输码即表示**同意接收**，杜绝被随意连接。码使用一次即失效。

### 4.3 文件分块传输协议

DataChannel 单条消息建议 ≤ 16KB（更稳），且必须等接收方回执做流控，避免缓冲爆满。约定一个简单应用层协议：

```
# 起始
{ kind: "meta", name, size, mime }
# 数据块（ArrayBuffer，offset 用于拼装与进度）
{ kind: "chunk", offset } + 二进制
# 接收方收到一块后回 ACK（流量控制 / 滑窗）
{ kind: "ack", offset }
# 结束
{ kind: "done" }
# 接收方校验完整性
{ kind: "ok" | "fail" }
```

- 发送方维护一个**滑动窗口**（如 4 块在途），收到 ack 再发下一批，兼顾吞吐与背压。
- 接收方按 `offset` 顺序写入 `Blob`（或 `FileSystemWritableFileStream` 落地大文件），收齐后触发下载。
- 进度条：前端按 `offset / size` 计算。

## 5. 实现拆解

### 5.1 后端：信令路由 `server/app/routers/rtc.py`

职责：WS 鉴权、在线池、邀请码撮合、SDP/ICE 转发。约 60–80 行。

核心结构：

```python
from fastapi import APIRouter, WebSocket, WebSocketDisconnect
from jose import JWTError

from app.utils.security import verify_token

router = APIRouter(prefix="/api/rtc")

# user_id -> WebSocket
online: dict[int, WebSocket] = {}
# 邀请码 -> {"from": user_id, "offer": sdp}
invites: dict[str, dict] = {}


async def _broadcast_online():
    payload = {"type": "presence", "users": [{"id": uid} for uid in online]}
    for ws in list(online.values()):
        await ws.send_json(payload)


@router.websocket("/signal")
async def signal(ws: WebSocket, token: str):
    # 1) 鉴权（复用现有 JWT）
    try:
        payload = verify_token(token)
        assert payload.get("type") == "access"
        uid = int(payload["sub"])
    except (JWTError, AssertionError, KeyError):
        await ws.close(code=4401)
        return

    await ws.accept()
    online[uid] = ws
    await _broadcast_online()

    try:
        while True:
            msg = await ws.receive_json()
            t = msg.get("type")

            if t == "invite_create":
                code = _gen_code()                       # 6 位短码
                invites[code] = {"from": uid, "username": user.username,
                                 "offer": msg["offer"]}
                await ws.send_json({"type": "invite_created", "code": code})

            elif t == "invite_join":
                rec = invites.pop(msg["code"], None)
                if rec:                                  # 把发起方 offer（含候选）下发
                    await ws.send_json({"type": "offer",
                                        "from": rec["from"],
                                        "username": rec["username"],
                                        "offer": rec["offer"]})
                else:
                    await ws.send_json({"type": "invite_not_found"})

            elif t == "answer":
                target = online.get(int(msg["to"]))
                if target:
                    # answer 同样自带候选；附带应答方用户名，让发起方知道对端是谁
                    await target.send_json({"type": "answer",
                                            "from": uid,
                                            "username": user.username,
                                            "answer": msg["answer"]})
                else:
                    await ws.send_json({"type": "peer_unavailable"})
            # 非 trickle：候选已裹进 offer/answer，不再有 candidate 分支。
    except WebSocketDisconnect:
        pass
    finally:
        online.pop(uid, None)
        await _broadcast_online()
```

> 注：在线用户名展示可在广播时按需 join 用户表，或前端复用已有用户信息接口。

在 `app/main.py` 注册：`app.include_router(rtc.router)`。

### 5.2 前端：传输页 `web/src/views/Transfer.vue`

职责：建立信令 WS、展示在线列表、生成 / 输入邀请码、发起 WebRTC、分块收发文件。

骨架（仅核心流程）：

```ts
// 信令：首帧发 token 鉴权（token 不走 URL，避免进 nginx 日志）
const ws = new WebSocket(`wss://${location.host}/api/rtc/signal`)
ws.onopen = () => ws.send(JSON.stringify({ token }))

const pc = new RTCPeerConnection({ iceServers: [] })
let dc: RTCDataChannel

// 等 ICE 收集完成，把候选裹进 localDescription（非 trickle 核心）
function waitIceComplete(p: RTCPeerConnection, timeoutMs = 4000) {
  return new Promise<void>((resolve) => {
    if (p.iceGatheringState === 'complete') return resolve()
    const done = () => resolve()
    p.addEventListener('icegatheringstatechange', () => {
      if (p.iceGatheringState === 'complete') done()
    })
    setTimeout(done, timeoutMs)   // 兜底，不无限等
  })
}

ws.onmessage = async (e) => {
  const m = JSON.parse(e.data)
  switch (m.type) {
    case 'presence':       onlineUsers.value = m.users; break
    case 'offer':          // 我是接收方：offer 已含 A 的候选
      pc.ondatachannel = (ev) => (dc = ev.channel)
      await pc.setRemoteDescription(m.offer)
      await pc.setLocalDescription(await pc.createAnswer())
      await waitIceComplete(pc)                       // answer 也裹好候选
      ws.send(JSON.stringify({ type: 'answer', to: m.from, answer: pc.localDescription }))
      break
    case 'answer':         // 我是发起方：answer 已含 B 的候选
      await pc.setRemoteDescription(m.answer); break
  }
}

// 发起：创建 DataChannel + offer，等候选裹好后拿邀请码
async function createInvite() {
  dc = pc.createDataChannel('file')
  await pc.setLocalDescription(await pc.createOffer())
  await waitIceComplete(pc)                           // offer 裹好候选
  ws.send(JSON.stringify({ type: 'invite_create', offer: pc.localDescription }))
}
// 收到 invite_created 后展示 code

// 接收方填码
ws.send(JSON.stringify({ type: 'invite_join', code }))
```

**文件收发**（分块发送、按 offset 拼装接收）：

```ts
const CHUNK = 16 * 1024

async function sendFile(file: File) {
  dc.send(JSON.stringify({ kind: 'meta', name: file.name, size: file.size, mime: file.type }))
  for (let offset = 0; offset < file.size; offset += CHUNK) {
    const buf = await file.slice(offset, offset + CHUNK).arrayBuffer()
    dc.send(buf)                 // 生产实现需加 ack 滑窗 + 背压
  }
  dc.send(JSON.stringify({ kind: 'done' }))
}

dc.onmessage = (e) => {
  // meta：建 Blob[]；chunk：push 并存 offset；done：合成 Blob 触发下载
}
```

> 生产实现需补：滑窗 + ack、错误重传策略、大文件用 `showSaveFilePicker` 流式落盘。

## 6. 安全考虑

| 风险 | 应对 |
|---|---|
| 陌生人接信令 | WS 握手校验 JWT（失败以 4401 关闭）；不在线、未认证一律拒绝。 |
| 被随意发起连接 | 邀请码一次性 + 接收方主动输入，等于显式同意。 |
| 明文传输 | WebRTC DataChannel 默认 DTLS 加密，端到端。 |
| 大文件占用浏览器内存 | 分块 + 滑窗 + 流式落盘（`FileSystemWritableFileStream`），避免整文件入内存。 |
| 邀请码爆破 | 6 位码、短时效（如 60s 过期）、尝试次数限制（可选）。 |

---

## 7. 限制与后续演进

- **跨网传输**：当前仅保证局域网纯 P2P；跨网需引入 STUN/TURN。可自托管 coturn 或用公共 STUN（如 `stun:stun.l.google.com:19302`）。
- **NAT 类型**：对称型 NAT 基本打不通，必须 TURN 兜底。
- **多对多 / 群传**：当前一对一；后续可扩展为多方 Mesh 或 SFU（超出本方案范围）。
- **断点续传**：基于 offset 协议可实现，需持久化传输进度。
- **移动端后台**：手机锁屏 / 切后台后 WebSocket 与 DataChannel 易被系统挂起，需提示用户保持页面在前台。
