# 新服务上线与网关扩展 SOP

> 日期：2026-07-19
> 适用：`airise-gateway` 网关托管模式下，接入一个**新的前后端分离项目**。
> 关联：架构 [`260719-nginx部署架构.md`](260719-nginx部署架构.md) · PixelPack 部署 [`../deploy.md`](../deploy.md)。

本文以一个虚构项目 `myapp`（域名 `myapp.airise.site`，后端容器 `myapp-api`，监听 8000）为例，演示从零接入的全过程。所有 `<project>` 占位符在你自己的项目里替换成实际项目名。

---

## 0. 接入清单（先读这个）

新项目要接入网关，本质是让**网关能按容器名找到后端 / 前端、能匹配域名**。清单：

| 项 | 由谁负责 | 关键值（以 myapp 为例） |
|---|---|---|
| 共享网络 `airise-web` | 全局一次性（已存在则跳过） | `docker network create airise-web` |
| 后端容器名 | 后端 compose | `myapp-api`，接入 `airise-web`，**不暴露宿主端口** |
| 前端交付方式 | 后端 compose / 宿主机 | **A** 宿主 `npm run build` 出 `/opt/myapp/web/dist`；**B** 项目自带 `myapp-web` 多阶段容器（见 §2） |
| 宿主数据目录 | 宿主机 | `/opt/myapp/data/uploads`（api 写；方式 A 另需 `/opt/myapp/web/dist`），属主按需 |
| DNS | DNS 服务商 | `myapp` A 记录 → 服务器 IP |
| 证书 | 宿主机 certbot | 通配 `*.airise.site` 已覆盖；无通配则单域签发 |
| 网关站点配置 | `airise-gateway/conf.d/` | 复制 `_template.example` → `myapp.airise.site.conf`，替换占位符（`location /`、`/uploads` 按交付方式 A/B 选） |
| 网关挂载 | `/opt/airise-gateway/docker-compose.yml` | **方式 A**：加 `web/dist` + `data/uploads` 卷；**方式 B**：零挂载（SPA、uploads 全由 web 容器提供） |
| 重载网关 | `/opt/airise-gateway` | 方式 A `docker compose up -d`（新挂载卷）；方式 B 仅 `nginx -s reload`（conf 卷挂载，无需 --build） |

---

## 1. 部署后端容器

### 1.1 compose 接入 `airise-web`

后端的 `docker-compose.yml` 必须：① 用固定 `container_name`（网关按此反代）；② 接入 external 网络 `airise-web`；③ **不对主机暴露端口**（外部一律走网关）。

```yaml
# /opt/myapp/docker-compose.yml
services:
  api:
    build: ./server
    image: myapp-api
    container_name: myapp-api          # ← 网关 upstream 就用这个名字
    restart: unless-stopped
    env_file:
      - ./server/.env
    environment:
      DATABASE_URL: "sqlite+aiosqlite:////app/data/data.db"
      UPLOAD_DIR: "/app/data/uploads"
    volumes:
      - ./data:/app/data
    networks:
      - airise-web                     # ← 共享网络，与网关按容器名互通

networks:
  airise-web:
    external: true                     # ← 必须 external，不要让 compose 自建
```

> 容器名即网关 upstream。改名要同步改网关 conf 的 `proxy_pass`，并 `--build` 网关。

### 1.2 数据目录与权限

```bash
mkdir -p /opt/myapp/data/uploads
# 若后端容器以非 root 用户(如 UID 1000)运行，宿主目录须 chown 对齐
chown -R 1000:1000 /opt/myapp/data
```

### 1.3 启动并自检

```bash
cd /opt/myapp
docker compose up -d --build
docker compose logs -f api
# 确认容器在网络里：
docker network inspect airise-web --format '{{range .Containers}}{{.Name}} {{end}}'
# 应能看到 myapp-api（以及 airise-gateway）
```

## 2. 构建前端（两种交付方式，二选一）

网关 `location /` 指向前端静态资源，有两种做法（详见架构文档 §3「前端 SPA 的两种交付方式」）：

**方式 A：网关直发宿主 dist**（本节示例，适合宿主机有 node、想免重建镜像）

```bash
cd /opt/myapp/web
npm install
npm run build        # → /opt/myapp/web/dist，稍后在网关 compose 里挂载（见 §4.2）
```

**方式 B：前端多阶段容器**（PixelPack 采用，适合前端全容器化）

项目 compose 自带 `myapp-web` 服务（`web/Dockerfile`：`node` 构建 → `nginx` serve），接入 `airise-web`，并只读挂载 uploads 目录由其 nginx 直发：

```yaml
  web:
    build: ./web
    image: myapp-web
    container_name: myapp-web
    restart: unless-stopped
    networks: [airise-web]
    volumes:
      - ./data/uploads:/app/data/uploads:ro   # api 写、web 只读直发 /uploads（同 compose 零漂移）
```

`web/nginx.conf` 内同时配置 SPA 与 `/uploads`：

```nginx
location /uploads/ { alias /app/data/uploads/; add_header Content-Disposition "inline"; }
location / { try_files $uri $uri/ /index.html; }
```

网关 `location /` 改为 `proxy_pass http://myapp-web;`（见 §4.1），`/uploads` 随 `location /` 转发给 web。**网关 compose 零挂载**（既不挂 dist 也不挂 uploads）。更新前端：`docker compose up -d --build web`。

> 无论哪种方式，前端服务（方式 B 的 `myapp-web`）**只处理静态资源**（SPA + `/uploads` 直发），`/api`、WS 一律由网关单层直连后端。`/uploads` 经 web 但纯静态，不涉及 WS/SSE 参数，不违反此约束。

## 3. DNS 与证书

- **DNS**：主机记录 `myapp` / 类型 `A` / 记录值=服务器公网 IP → `myapp.airise.site`。
  验证：`dig +short myapp.airise.site`。
- **证书**：
  - 已有通配 `*.airise.site` → 直接复用，conf 里证书路径填 `/etc/letsencrypt/live/airise.site/`，**无需额外操作**。
  - 无通配 → 单域签发，conf 证书路径改为 `/etc/letsencrypt/live/myapp.airise.site/`（签发见架构文档 §8）。

## 4. 网关扩展（核心）

网关配置变更 = 改 `nginx/conf.d/` 或 `snippets/`（卷挂载，`reload` 生效）+ 改 `docker-compose.yml` 挂载（`up -d` 重建容器）。两步：

### 4.1 由模板生成站点配置

复制模板，**另存为 `*.conf` 文件名**（不要直接改 `_template.example`；它不是 `.conf`，nginx 不加载，留作参考）：

```bash
cd /opt/airise-gateway
cp conf.d/_template.example conf.d/myapp.airise.site.conf
```

把新文件里的占位符全部替换：

| 占位符 | 替换为 | 说明 |
|---|---|---|
| `<project>.airise.site`（两处：443/80 的 `server_name`） | `myapp.airise.site` | 域名 |
| `root  /var/www/<project>;` | `root  /var/www/myapp;` | SPA 根（须与 §4.3 挂载目标一致） |
| `proxy_pass http://<project>-api:8000;` | `proxy_pass http://myapp-api:8000;` | 后端容器名 + 端口 |
| `alias /var/www/<project>-uploads/;` | `alias /var/www/myapp-uploads/;` | 上传目录（须与 §4.3 挂载目标一致） |

替换后核心片段应类似：

```nginx
server {
    listen 443 ssl;
    http2 on;
    server_name myapp.airise.site;

    ssl_certificate     /etc/letsencrypt/live/airise.site/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/airise.site/privkey.pem;
    ssl_protocols TLSv1.2 TLSv1.3;

    root  /var/www/myapp;
    index index.html;
    client_max_body_size 25m;

    location /api/ {
        proxy_pass http://myapp-api:8000;     # 容器名，需同在 airise-web
        include /etc/nginx/snippets/proxy.conf;
    }

    location /uploads/ {
        alias /var/www/myapp-uploads/;
        add_header Content-Disposition "inline";
    }

    location / {
        try_files $uri $uri/ /index.html;     # SPA history 回退；非 SPA 项目删掉
    }
}

server {
    listen 80;
    server_name myapp.airise.site;
    return 301 https://$host$request_uri;
}
```

要点：
- **每个 `proxy_pass`（后端 api）的 location 必须 `include /etc/nginx/snippets/proxy.conf;`**：统一 WS 升级、头透传、关缓冲、长超时。漏 include 会导致 SSE/WS 卡住。注意：反代到前端静态容器（方式 B 的 `location /`）**不要** include 该片段（静态资源需要默认缓冲，关了反而低效）。
- 后端路径不是 `/api/`（如直接 `/`、`/v1/`）就按实际改 `location`。
- `$connection_upgrade` 由全局唯一的 `00-upgrade-map.conf` 提供，**不要在每个站点重复定义 map**。
- **交付方式 B（前端容器）**：上面 `root` / `index` / `location /uploads/` / `location / { try_files ... }` 改为
  ```nginx
  location / {
      proxy_pass http://myapp-web;
      proxy_set_header Host $host;
      proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
  }
  ```
  删去 `root` / `index` 与整个 `location /uploads/` 块（`/uploads` 随 `location /` 转发给 web 容器，由其 nginx 直发）。§4.2 网关 compose **零挂载**（dist、uploads 都由 `myapp-web` 容器提供）。

### 4.2 给网关 compose 增加挂载（仅方式 A）

**方式 B 跳过本节** —— 网关不挂任何项目路径。仅方式 A（网关直发）需编辑 `/opt/airise-gateway/docker-compose.yml`，在 `volumes:` 增加该项目的 dist + uploads（只读）：

```yaml
    volumes:
      - /etc/letsencrypt:/etc/letsencrypt:ro
      # ……既有方式 A 项目的挂载保留……（PixelPack 是方式 B，网关零挂载）
      # ▼ 新增 myapp（方式 A）
      - /opt/myapp/web/dist:/var/www/myapp:ro
      - /opt/myapp/data/uploads:/var/www/myapp-uploads:ro
```

> 方式 A 挂载目标路径（`/var/www/myapp`、`/var/www/myapp-uploads`）必须与 §4.1 conf 里的 `root` / `alias` 完全一致，否则 404；且网关与 myapp 是两个 compose，硬编码绝对路径易漂移，换部署目录要同步改。方式 B 无此问题。

### 4.3 重载网关

conf 是卷挂载，改完 `reload` 即可；仅方式 A 新增了挂载卷时才需 `up -d` 重建容器：

```bash
cd /opt/airise-gateway
docker compose up -d                  # 仅方式 A 新增挂载卷时需要（方式 B 可跳过）
docker exec airise-gateway nginx -t   # 配置语法自检
docker exec airise-gateway nginx -s reload
curl -I https://myapp.airise.site     # 应 200/301（看 SPA 首页）
```

## 6. 速查：网关侧改动的情形

| 改动 | 操作 |
|---|---|
| 接入方式 B 新项目（加 `conf.d/<proj>.airise.site.conf`，无挂载卷） | `docker exec airise-gateway nginx -s reload` |
| 接入方式 A 新项目（加 conf + compose 新增挂载卷） | `docker compose up -d`（让新挂载卷生效）→ `nginx -s reload` |
| 改某站点配置（域名/路径/证书路径，不动 compose） | `docker exec airise-gateway nginx -s reload` |
| 仅证书续期（文件内容变、配置未变） | `docker exec airise-gateway nginx -s reload` |

> 卷挂载模式：conf 是宿主机文件直接挂进容器，改完 `reload` 即生效，**不再需要 `--build`**。

## 7. 常见坑

- **网关启动报 `host not found in upstream "xxx-api"`**：后端容器名拼错、或后端容器没接入 `airise-web`、或 `airise-web` 没建。nginx 启动时解析不了 upstream 主机名会直接 emerg 退出（网关无限重启）。也可能是 `conf.d/` 里有带占位符的模板被当 `.conf` 加载——模板必须是 `_template.example`（非 `.conf`）。
- **404 静态 / 上传**：方式 A —— compose 挂载目标路径与 conf 的 `root`/`alias` 不一致，或网关与项目两个 compose 的宿主绝对路径漂移（换部署目录没同步改）。方式 B —— web 容器没挂 `./data/uploads`，或 `web/nginx.conf` 缺 `location /uploads/`。
- **SSE / WS 卡 pending**：该 location 漏了 `include /etc/nginx/snippets/proxy.conf;`（少了 WS 升级头 + `proxy_buffering off`）。
- **上传目录写不进**：后端容器非 root 用户时，宿主 `data/uploads` 属主未对齐（如非 1000）。
- **80 → 443 不跳转**：漏了 80 端口的 `return 301` server 块。
- **改了 conf 却没生效**：忘了 `docker exec airise-gateway nginx -s reload`。
