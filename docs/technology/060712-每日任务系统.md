# PixelPack · 每日任务系统 V1.0 重构方案

> 日期：2026-07-12
> 模块：任务系统（Quest → Task 重构） / 对话生成 / 等级经验 / 成就
> 运行环境：**GLM（`https://open.bigmodel.cn/api/anthropic`）+ `claude-agent-sdk`**（复用 `intel` 模块已验证模式，见《260711-ClaudeCode接入.md》）
> 目标：把当前「硬编码 + 随机抽样」的固定任务系统，重构为「**自然语言对话生成 + 手动自定义 + 等级经验联动 + 任务核心成就**」的可成长任务系统。

---

## 0. 需求来源

```text
## V1.0：
- 用接入的 Claude Code，通过自然语言描述学习/工作计划，生成每日需执行的任务清单
- 可能涉及 function call 调数据库写入
- 前后端以对话交互，聊天对话需记录到数据库
- 可能用 SSE 做打字机流式输出
- 任务系统同时支持手动自定义添加，不依赖模型
- 联动人物等级/经验：完成每日任务增加
- 联动长期成就系统：完成指定数量任务解锁成就
- 先实现任务系统核心功能

## V2.0（本次只做扩展预留，不实现）：
- 定时任务：每天定时推送待执行任务
- 每天结束时整理今天完成的任务
```

### 本次决策（已与产品确认）

| 维度 | 决策 | 说明 |
|------|------|------|
| 任务模型 | **完全替换** | 废弃 `QUEST_DEFS` 随机抽样；新建 `tasks` 表，任务来源仅 AI 生成 + 手动添加 |
| AI 写入方式 | **Agent 直接写库** | Agent 在对话中通过 `create_task` 工具（function call）直接落库，SSE 流式展示过程，任务即时进入清单 |
| 范围 | **仅 V1 核心** | NL 生成 + 对话(入库) + SSE 打字机 + 手动添加 + 等级经验 + 成就联动。V2 定时推送/复盘只留扩展点 |
| 成就 | **围绕任务重新设计** | 改为以「任务完成 / 打卡」为核心的成就，替换现有 12 个以物品为核心的成就 |

## 1. 现状（已读懂的逻辑）

### 1.1 后端
- 技术栈：FastAPI + SQLAlchemy 2.0 async + SQLite(aiosqlite) + JWT。
- 任务代码叫 `Quest`：`models/quest.py` 两张表 `daily_quests` / `user_achievements`；任务定义 `QUEST_DEFS`（5 种，如 `ADD_ITEMS`/`VIEW_ITEMS`）与成就 `ACHIEVEMENT_DEFS`（12 种）**硬编码在 `services/quest.py`**，每天 `random.sample` 抽 3 个。
- 完成链路：`increment_quest_progress` → 满额置 `completed` → `_add_quest_exp` 加经验 → 写 `quest_event` 系统日志 → `check_achievements` 判定解锁。
- 进度主要由物品模块自动埋点（`routers/items.py`），部分 key 靠前端诚实上报 `POST /api/quests/progress`。
- **无 Alembic**，启动时 `Base.metadata.create_all` 建表（**不会自动 ALTER 已存在表**，加列需手写迁移）。

### 1.2 已发现的两处缺陷（本方案一并修复）
- **R-A 经验不持久化**：`User` 没有独立 level/exp 列，经验存在 `User.settings` JSON 的 `quest_exp`/`daily_completed`；该 JSON 列未用 `MutableDict`，SQLAlchemy **检测不到原地修改**。实测 `users.settings` 全为 `{}`，经验从未真正写入。
- **R-B 成就死代码**：`WARRANTY_WATCHER` 在 `ACHIEVEMENT_DEFS` 有定义但 `check_achievements` 无判定条件 → 永不解锁。

### 1.3 Claude Code 接入（`intel` 模块，已端到端验证）
- 协议层 `claude-agent-sdk`，模型层 GLM `glm-5.2`，子进程继承 `ANTHROPIC_*` 环境变量。
- 自定义 in-process MCP 工具（`@tool` + `create_sdk_mcp_server`）替代 GLM 不可用的 WebSearch/WebFetch；`output_format` 结构化输出实测可用；`async for msg in query(...)` 流式已用于打日志（**目前非 SSE，本方案首次引入面向前端的 SSE**）。
- APScheduler 每日定时跑一次。

### 1.4 前端
- `Quests.vue` / `Dashboard.vue` 两处均为**只读看板**，无「完成/添加/勾选」交互。
- **聊天 UI / SSE / 打字机 = 0 实现**（`WorldMap.vue` 仅有一段纯前端 `setTimeout` 打字氛围装饰，不可复用为 SSE 客户端）。
- 设计 token 成熟：`--pixel-*` + `.pixel-border` + `var(--font-pixel)`；`design/260712-dailytask/` 已建好但为空（front-design skill 产出位置）。
- MainLayout 顶导航**没有「任务」入口**，需补。

## 2. 整体架构

```
┌──────────────────────────────────────────────────────────────────┐
│                         FastAPI (server/app)                      │
│                                                                   │
│  ┌─────────────── 对话生成路径（流式 SSE）──────────────────┐     │
│  │ POST /api/chat/sessions/{id}/messages   (text/event-stream) │     │
│  │   1. 用户消息入 chat_messages                               │     │
│  │   2. services/task_agent.run(session, user_msg)             │     │
│  │      claude-agent-sdk → GLM，带 in-process MCP 工具:         │     │
│  │      ├─ list_today_tasks()   读上下文(防重复)                │     │
│  │      ├─ create_task(...)     ★ function call 直接写 tasks 表│     │
│  │      └─ get_user_profile()   等级/偏好上下文                 │     │
│  │   3. 流式 AssistantMessage.TextBlock 增量 → SSE event 'delta'│     │
│  │      工具调用 → SSE event 'tool' / 'task_created'            │     │
│  │   4. 结束后 assistant 整条消息入 chat_messages               │     │
│  └─────────────────────────────────────────────────────────────┘     │
│                                                                   │
│  ┌─────────────── 任务 CRUD 路径（REST）────────────────────┐      │
│  │ GET/POST/PATCH/DELETE /api/tasks          手动增删改查    │      │
│  │ POST /api/tasks/{id}/complete             完成→加经验→成就 │      │
│  │ GET  /api/quests/summary | /achievements  等级/成就聚合   │      │
│  └──────────────────────────────────────────────────────────┘      │
│                                                                   │
│  storage: SQLite  tasks / chat_sessions / chat_messages /          │
│           user_achievements  +  users.exp（新增列）                │
└─────────────────────────────────────────────────────────────────────┘
              ▲ ofetch(REST)  ▲ fetch+ReadableStream(SSE,带 JWT)
              │              │
        web/src/api/tasks.ts  web/src/utils/sse.ts
              │              │
        views/Tasks.vue   views/Chat.vue（对话 + 打字机）
```

**数据流要点**
1. **生成路径**（低频、可慢）：用户描述 → SSE 流式 → Agent 边想边调 `create_task` 直接写库 → 前端清单实时新增。
2. **执行路径**（高频、必须快）：用户在任务页勾选完成 → REST 同步加经验 + 判成就，毫秒级。
3. **读取路径**：任务清单/等级/成就纯 DB 查询，绝不触发 Agent。
4. **失败隔离**：Agent 失败不影响已有任务的读取与完成；对话失败只丢本次回复，历史不丢。

## 3. 数据模型

### 3.1 新增 `Task`（`tasks` 表）—— 替换 `daily_quests`

```
Task（用户任务）
├── id              INTEGER PK autoincrement
├── user_id         INTEGER  (索引)
├── title           VARCHAR(120)  任务标题
├── description     TEXT          描述/备注（可空）
├── category        VARCHAR(32)   分类: study/work/life/health/other（索引）
├── source          VARCHAR(16)   来源: ai / manual
├── target          INTEGER       目标值，默认 1（支持多步，如"做3道题"=3）
├── progress        INTEGER       当前进度，默认 0
├── completed       BOOLEAN       是否完成，默认 False
├── completed_at    DATETIME      完成时间（可空）
├── exp_reward      INTEGER       经验奖励（AI 生成时由 Agent 建议区间，手动添加时可选/默认）
├── due_date        DATE          任务日期（每日清单 = due_date == today）（索引）
├── recurrence      VARCHAR(16)   重复: once / daily / weekly（V1 先落字段，daily 由 V2 调度消费）
├── sort_order      INTEGER       列表排序，默认 0
├── created_at      DATETIME
└── updated_at      DATETIME
```

**设计说明**
- 「每日任务」语义 = `due_date == today`；不再有独立的 `quest_date` 随机抽样表。
- `target>1` 时为多步任务，`progress` 推进到 `target` 即完成；`target=1` 退化为勾选。
- `source` 区分 AI 与手动，供成就（`AI_EXPLORER` / `SELF_MADE`）与统计用。
- `recurrence` 本版只写不消费（V2 定时任务据此每天实例化），避免过度设计。
- 旧 `daily_quests` 表**保留不删**（避免丢历史），代码不再读写；如需清理提供一次性脚本。

### 3.2 新增 `ChatSession` / `ChatMessage`（对话入库）

```
ChatSession（会话）
├── id           INTEGER PK
├── user_id      INTEGER  (索引)
├── title        VARCHAR(120)  首条用户消息截断生成
├── created_at   DATETIME
└── updated_at   DATETIME

ChatMessage（消息）
├── id           INTEGER PK
├── session_id   INTEGER  (索引, FK → chat_sessions.id)
├── role         VARCHAR(16)   user / assistant
├── content      TEXT          消息文本（assistant 为流式结束后的整条）
├── meta         JSON          可选: {tools:[create_task x3], tasks_created:[ids], turns}
└── created_at   DATETIME
```

**说明**：流式过程中的 token 增量**不入库**，只在 SSE 通道传输；消息**整条**在流结束后落库，保证历史可回放、低成本。

### 3.3 `User` 加列 —— 修复经验持久化（R-A）

`models/user.py` 新增列：

```
User
├── exp           INTEGER  累计经验，默认 0     ← 新增
└── settings      JSON      （保留，不再用于存 exp）
```

**⚠️ 关键**：`create_all` 不会 ALTER 已存在的 `users` 表。必须在 `main.py` lifespan 里加**幂等迁移**：

```python
# main.py lifespan，create_all 之后
async with engine.begin() as conn:
    await conn.execute(text(
        "ALTER TABLE users ADD COLUMN exp INTEGER DEFAULT 0"
    ))  # 用 PRAGMA/try-except 包成幂等；SQLite 重复 ADD 会报错
```

封装为 `utils/migrate.py` 的 `ensure_column(conn, table, col, ddl)`，按 `PRAGMA table_info` 检测后补列，幂等可重入。

**等级公式（重新定义，以任务为核心）**：
```
level = exp // 50 + 1            # 每 50 经验一级
exp_to_next = 50                 # 固定步长（可改分段表，V1 先线性）
```
- 经验**仅来自**：完成任务（`exp_reward`）+ 解锁成就（成就自带 `exp`）。
- 物品不再直接折算经验（物品已有「日均成本」等统计展示，不靠经验）。移除旧 `get_user_level` 里的 `item_count*5` 项。

### 3.4 成就（`user_achievements` 表不变；`ACHIEVEMENT_DEFS` 重新设计）

`user_achievements`（已存在）继续用作解锁记录。`ACHIEVEMENT_DEFS` 内容**改为任务核心**（仍代码常量，不迁 DB）：

| key | 名称 | 判定 | exp |
|-----|------|------|-----|
| `FIRST_TASK` | 初试身手 | 累计完成 ≥1 任务 | 20 |
| `TASKS_10` | 坚持十连 | 累计完成 ≥10 | 40 |
| `TASKS_50` | 半百里程碑 | 累计完成 ≥50 | 100 |
| `TASKS_100` | 百炼成钢 | 累计完成 ≥100 | 200 |
| `STREAK_3` | 三日不辍 | 连续打卡 ≥3 天 | 40 |
| `STREAK_7` | 一周不缺 | 连续打卡 ≥7 天 | 80 |
| `STREAK_30` | 月度勤勉 | 连续打卡 ≥30 天 | 240 |
| `PERFECT_DAY` | 完美一天 | 单日清单全部完成 | 60 |
| `CATEGORY_SCHOLAR` | 学者 | `study` 类累计 ≥30 | 120 |
| `AI_EXPLORER` | 智者辅佐 | AI 生成并完成 ≥10 | 80 |
| `SELF_MADE` | 自律工匠 | 手动添加并完成 ≥10 | 80 |
| `CENTURY` | 百日征途 | 累计打卡 ≥100 天 | 500 |

**判定时机**：每次 `complete_task` 后聚合 `tasks` 表统计（`count(completed)`、`source` 分布、`group by due_date` 算连续打卡），对照阈值解锁。
**连续打卡定义**：`due_date` 当天至少完成 1 个任务算「打卡一天」；从今天回溯连续无断档的最大天数。

## 4. 任务 CRUD 与完成联动

### 4.1 REST 端点（新增 `routers/tasks.py`，`/api/tasks`）

| 方法 | 路径 | 说明 |
|------|------|------|
| `GET` | `/api/tasks?date=today` | 当日清单（默认 today，可传 YYYY-MM-DD） |
| `POST` | `/api/tasks` | 手动添加（强制 `source=manual`） |
| `PATCH` | `/api/tasks/{id}` | 编辑标题/描述/分类/目标/经验/日期 |
| `DELETE` | `/api/tasks/{id}` | 删除（未完成不退经验；完成的不允许直接删，需先 uncomplete） |
| `POST` | `/api/tasks/{id}/complete` | 标记完成 → **加经验 → 判成就 → 返回升级/解锁信息** |
| `POST` | `/api/tasks/{id}/uncomplete` | 撤销完成 → 扣回经验（幂等：仅当当前 completed=True） |
| `POST` | `/api/tasks/{id}/progress` | 多步任务推进 `{increment}`（满额自动 complete） |

**保留并改造**（`routers/quests.py`）：
- `GET /api/quests/summary` → 返回 `{ level, exp, exp_to_next, today: {total, completed}, tasks_total, achievements:[...], streak }`
- `GET /api/quests/achievements` → 返回成就含解锁状态
- **删除** `GET /api/quests/daily` 与 `POST /api/quests/progress`（旧随机抽样逻辑），前端改用 `/api/tasks`。

### 4.2 完成链路（核心）

```python
# services/task.py
async def complete_task(db, user, task) -> CompleteResult:
    if task.completed:
        return CompleteResult(noop=True)
    task.completed = True
    task.progress = task.target
    task.completed_at = now()

    user.exp += task.exp_reward                       # 独立列，修复 R-A
    new_level = user.exp // 50 + 1
    leveled_up = new_level > old_level

    await journal.create_system_journal(db, user.id,
        category="task_event", icon="▣",
        title=f"完成任务: {task.title}", exp=task.exp_reward)
    if leveled_up:
        await journal.create_system_journal(..., category="level_up",
            title=f"等级提升 → Lv.{new_level}")

    unlocked = await check_achievements(db, user)     # 任务核心判定
    await db.commit()
    return CompleteResult(exp_gained=task.exp_reward, level=new_level,
                          leveled_up=leveled_up, achievements_unlocked=unlocked)
```

- 物品模块旧的 `increment_quest_progress` 埋点（`routers/items.py`）**移除**，物品不再驱动任务系统。

## 5. Claude Code Agent：自然语言生成 + function call 写库

复用 `intel` 模块模式，新增 `services/task_agent.py`。

### 5.1 in-process MCP 工具（替代/补充，核心是写库）

```python
# services/task_agent.py
from claude_agent_sdk import tool, create_sdk_mcp_server

@tool(name="list_today_tasks", description="返回当前用户今日已有任务清单(标题/分类/状态)，避免重复创建。",
      input_schema={})
async def list_today_tasks(args):
    rows = await task_svc.list_today_raw(user_ctx)        # 由注入的 user_ctx 取 user_id
    return {"content": [{"type": "text", "text": json.dumps(rows, ensure_ascii=False)}]}

@tool(name="create_task",
      description="""为用户今日清单创建一个任务并立即写入数据库。
参数: title(必填,简洁可执行), description(可选), category(study/work/life/health/other,默认study),
target(目标步数,默认1), exp_reward(经验,简单任务10/中等20/困难30,默认按难度), due_date(可选,默认今天)。""",
      input_schema={"title": str, "description": str, "category": str,
                    "target": int, "exp_reward": int, "due_date": str})
async def create_task(args):
    task = await task_svc.create_task(db_ctx, user_ctx, source="ai", **args)  # ★ 直接写库
    return {"content": [{"type": "text", "text": json.dumps({"id": task.id, "title": task.title})}]}
```

- 工具返回值遵循 MCP 内容块约定 `{"content":[{"type":"text","text":...}]}`（裸 dict 会被判 no output，见《ClaudeCode接入》§3 实测）。
- **user_id / db 注入**：`run()` 内用闭包/`functools.partial` 把当前用户和 session 工厂绑进工具 handler，工具不暴露 user_id 给模型（防越权）。
- 注册：`create_sdk_mcp_server(name="taskkit", tools=[list_today_tasks, create_task])`。

### 5.2 Options / Prompt

```python
def build_options() -> ClaudeAgentOptions:
    return ClaudeAgentOptions(
        mcp_servers={"taskkit": TASKKIT_SERVER},
        allowed_tools=["list_today_tasks", "create_task"],
        permission_mode="bypassPermissions",
        max_turns=settings.TASK_AGENT_MAX_TURNS,   # ~10
        system_prompt=SYSTEM_PROMPT,
        # 不设 output_format：任务靠工具写库，最后让 Agent 用自然语言总结即可（喂给 SSE）
    )

SYSTEM_PROMPT = """你是 PixelPack 的每日任务规划助手。用户会用自然语言描述今天的学习/工作计划，
你要把它拆解成 3-8 个**具体、可执行、可勾选**的今日任务。
规则：
1. 先调用 list_today_tasks 看已有任务，避免重复。
2. 对每个拆解出的任务，调用 create_task 写入数据库（不要只口头列出）。
3. 经验按难度给：简单 10、中等 20、困难 30。
4. category 从 study/work/life/health/other 选最贴切的。
5. 标题用动宾短语、可执行（如「读完《XX》第2章并写3条笔记」而非「学习」）。
6. 全部创建后，用一段话总结今天排了哪些任务、总经验、建议从哪个开始。"""
```

### 5.3 流式调用 + SSE 事件映射

```python
async def stream_run(session_id, user_msg):
    # 1. 用户消息入库
    await chat_svc.add_message(session_id, role="user", content=user_msg)
    async for msg in query(prompt=user_msg, options=build_options()):
        if isinstance(msg, AssistantMessage):
            for block in msg.content:
                if isinstance(block, TextBlock):
                    yield {"type": "delta", "text": block.text}        # 打字机增量
                elif isinstance(block, ToolUseBlock):
                    yield {"type": "tool", "name": block.name}         # 「正在创建任务…」
        elif isinstance(msg, UserMessage):
            for block in msg.content:
                if isinstance(block, ToolResultBlock):
                    if ctx.last_tool == "create_task":
                        yield {"type": "task_created", "task": parse_task(block)}  # 清单即时刷新
    # 2. 拼接 assistant 整条文本，入库
    await chat_svc.add_message(session_id, role="assistant", content=full_text, meta={...})
    yield {"type": "done"}
```

> **实测注意**：`claude-agent-sdk` 的流式以「消息」为单位产出（`AssistantMessage` 可能整段），未必逐 token。打字机效果可由前端对 `delta` 文本再做逐字 `setInterval` 渲染（参考 `WorldMap.vue` 的 `startTicker` 思路），保证视觉上的打字感，不强依赖 SDK 逐 token 推送。

## 6. SSE 端点（新增 `routers/chat.py`）

```
对话
  POST   /api/chat/sessions                 新建会话 → {id}
  GET    /api/chat/sessions                 会话列表
  GET    /api/chat/sessions/{id}/messages   历史消息
  DELETE /api/chat/sessions/{id}            删除会话
  POST   /api/chat/sessions/{id}/messages   ★ SSE 流式 (text/event-stream)
       body: { content: string }
       events:
         data: {"type":"delta","text":"..."}      token/段增量
         data: {"type":"tool","name":"create_task"}
         data: {"type":"task_created","task":{...}}
         data: {"type":"done"}
```

- 用 FastAPI `StreamingResponse(generator, media_type="text/event-stream")`，`Cache-Control: no-cache`，`X-Accel-Buffering: no`。
- 鉴权：全部 `Depends(get_current_user)`，且校验 session 归属当前用户。
- **并发保护**：`asyncio.Lock` per session（同一会话不并发跑两个 Agent 子进程），全局 `max_turns` 兜底。
- 客户端**不能用 `EventSource`**（无法带 `Authorization` 头）→ 用 `fetch` + `ReadableStream` 解析，封装在 `web/src/utils/sse.ts`。

### 7.2 SSE 客户端（关键）

```ts
// utils/sse.ts —— 带 JWT 的 SSE
export async function postSSE(url: string, body: any, onEvent: (e: SSEEvent) => void) {
  const token = localStorage.getItem('access_token')
  const res = await fetch('/api' + url, {
    method: 'POST',
    headers: { 'Content-Type': 'application/json', Authorization: `Bearer ${token}` },
    body: JSON.stringify(body),
  })
  const reader = res.body!.getReader()
  const decoder = new TextDecoder()
  let buf = ''
  while (true) {
    const { done, value } = await reader.read()
    if (done) break
    buf += decoder.decode(value, { stream: true })
    let idx
    while ((idx = buf.indexOf('\n\n')) >= 0) {           // SSE 事件以空行分隔
      const raw = buf.slice(0, idx); buf = buf.slice(idx + 2)
      const data = raw.split('\n').filter(l => l.startsWith('data:')).map(l => l.slice(5).trim()).join('')
      if (data) onEvent(JSON.parse(data))
    }
  }
}
```

401 刷新逻辑沿用 `api/client.ts` 的 `tryRefresh`（SSE 首次响应若 401 → 刷新重发一次）。

## 12. 风险与注意事项

| 级别 | 项 | 应对 |
|------|----|------|
| 🔴 R1 | **GLM 流式粒度** | SDK 可能按「消息」而非「token」产出，打字机感由前端逐字渲染补足；不强依赖逐 token。 |
| 🔴 R2 | **SQLite 加列** | `create_all` 不 ALTER 已存表；必须 `ensure_column` 幂等迁移，否则 `users.exp` 永远不存在（R-A 复现）。 |
| 🟠 R3 | **Agent 写库越权** | 工具 handler 内部绑定 user_id，不把 user_id 暴露给模型；`create_task` 只能写当前用户。 |
| 🟠 R4 | **SSE 鉴权** | 必须用 fetch+ReadableStream（带 Authorization），不可用 EventSource（不能带头）；401 沿用刷新逻辑。 |
| 🟠 R5 | **并发与成本** | per-session `asyncio.Lock` 防同会话并发；`max_turns`/prompt 限制单次最多 ~8 个任务，控制 GLM token。 |
| 🟡 R6 | **经验回滚一致性** | `uncomplete` 需扣回经验并尽量回滚成就（V1 简化：扣经验，成就不回收，仅记日志），文档明确语义。 |
| 🟡 R7 | **历史数据** | 旧 `daily_quests` 保留不删；移除物品埋点后，旧物品类成就失效（已与「围绕任务重新设计」决策一致）。 |
| 🟡 R8 | **`.env` 含明文 GLM token**（已提交仓库） | 安全隐患，建议迁出 `.env`、改由部署环境注入并清理 git 历史（独立处理，不阻塞本方案）。 |
