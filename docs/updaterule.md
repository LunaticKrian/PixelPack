# 仓库更新规则（SOP）

> 适用范围：**大型 feature / 较大改动**。
> 小修（typo、单行 bugfix、文案调整、纯依赖升级）不必走全流程，直接改即可，避免流程过重。

---

## 0. 命名规范

日期前缀统一 `YYMMDD`（例：`260719` = 2026-07-19）。

| 对象 | 规范 | 示例 |
|---|---|---|
| 分支 | `<type>/YYMMDD-<英文slug>` | `feat/260719-payment` |
| 设计稿目录 | `design/YYMMDD-<英文slug>/` | `design/260718-p2ptransport/` |
| 技术方案文档 | `docs/technology/YYMMDD-<中文标题>.md` | `docs/technology/260718-点对点数据传输.md` |

- **分支 `<type>`**：`feat/`（功能）、`ui/`（纯界面重构）、`fix/`（修 bug）、`docs/`（文档）。
- **分支名一律英文 slug**，不要用中文，避免在 shell / CI / git hook 里转义出错。
- 设计稿目录沿用英文 slug，技术方案文档沿用中文标题（与历史一致）。

---

## 1. 拉取分支

按命名规范从 `main` 拉新分支：

```bash
git checkout main && git pull
git checkout -b feat/260719-payment
```

后续所有改动都在该分支上进行，完成后再合回 `main`。

---

## 2. 涉及前端 → 先出设计稿

若本次改动包含前端新增或修改，**在写业务代码之前**先产出高保真设计稿到：

```
design/YYMMDD-<英文slug>/
├── *.html      页面结构
├── styles.css  设计系统（沿用全局 token --pixel-*）
└── main.js     （可选）交互/状态预览
```

设计稿须复用现有视觉体系（冷调深蓝像素 + CRT 复古、`--pixel-*` token、Press Start 2P / Ark Pixel / VT323 字体、硬边框无圆角、steps() 阶跃动画），并签名一个**有辨识度的核心元素**，不要落入通用模板默认值。

> 纯后端 / 无界面改动的特性可跳过本步。

---

## 3. 输出技术方案

在 `docs/technology/YYMMDD-<中文标题>.md` 写技术方案，至少覆盖：

- 背景与目标、选型结论（为什么选这个方案、否决了什么）
- 总体架构 / 数据流 / 接口
- 关键约束与前置条件（如安全上下文、网络、依赖服务）
- 涉及改动清单（哪些文件新增 / 修改）

**设计稿与技术方案必须互相校对**：界面状态、交互、数据展示要与接口/数据流一一对应，不能各说各话。

> 顺序由特性决定：UI 驱动型先设计稿后方案；架构/后端驱动型先方案后设计稿。两者都须在实现前完成。

---

## 4. Review 技术方案

Review 分两层：

1. **自查（我来做）**
   - 技术正确性（并发、状态机、错误处理、边界）
   - 是否落入了「任何相似需求都会产出的通用默认」（如奶油底+衬线、纯黑+酸绿、报纸三栏这种 AI 默认皮）——若有，改掉并说明
   - 安全 / 鉴权 / 数据持久化 / 部署影响
2. **有疑问随时 ask user**：不要在不确认的前提下闷头实现。
3. **用户确认**：由你最终拍板前端 UI 和技术方案是否 OK。

> 只有双方都确认无误后才进入实现。历史经验：这一步能抓到真 bug（如 WebRTC trickle ICE 的候选寻址竞态）。

---

## 5. 实现技术方案

按确认后的方案落地。实现过程中若**偏离了原方案**（换了实现思路、砍了/加了功能），必须**回头更新技术方案文档**，保持文档与代码一致——不要让文档和实现各走各的。

---

## 6. 完成定义（Definition of Done）

实现完成后，逐项确认：

- [ ] 前端 `npm run build`（含 `vue-tsc` 类型检查）通过
- [ ] 后端语法 / 导入正常，涉及路由已注册
- [ ] 若涉及部署（Docker / nginx / 环境变量）→ 同步更新 `docs/deploy.md`
- [ ] 设计稿与最终实现的一致性回检（如有偏差，回写设计稿或文档）
- [ ] 提交 commit（按需推分支 / 发起合并）

---

## 流程速览

```
拉分支 → [涉前端?] 出设计稿 → 出技术方案 → review(自查+ask+你确认) → 实现 → DoD 收尾 → 提交
```
